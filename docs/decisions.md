# Decisions and reflection

Короткий пост-мортем по выбору примитивов, сделанным компромиссам и тому, что бы доделали при бесконечном времени.

Часть секций (что не сработало, surprises) фиксируется по итогам реального прогона на полигоне `process/deepseek-client/` (Phase 6). На момент написания этого документа выполнены Phases 0–5: scaffolding, sub-agents, slash-commands, skills, state-guard hook (с тестами), документация. Phases 6–7 (прогон + опциональная имплементация) — впереди.

Структура документа:

1. **Что не сработало с первого раза** — заполняется после прогона; сейчас заготовлены ожидаемые категории.
2. **Trade-offs** — сделанные сейчас, на этапе конфигурации.
3. **Что бы сделали при бесконечном времени** — backlog.
4. **Surprises** — заполняется после прогона.
5. **Сознательные компромиссы** — то, что вынесено из scope намеренно.

## Trade-offs, которые мы приняли

Эти решения зафиксированы на момент сдачи Phase 5. Каждое — компромисс, не «правильный ответ».

- **state-guard как хук + инлайн-валидация в командах (belt-and-suspenders).** Не чисто-хук и не чисто-инлайн. Причина: хук демонстрирует примитив (это часть оценки задания), инлайн делает команду самодостаточной и устойчивой к misconfiguration хука на машине ревьюера. Дублирование оправдано — оба слоя дешёвы.
- **Skills как конвенции, а не как обёртки над шаблонами.** Шаблоны живут в `docs/templates/` — единственный канонический источник. Skills в `.claude/skills/` ссылаются на них и добавляют дисциплину (рубрику глубины, обязательные секции, banned-фразы). Альтернатива — продублировать тексты шаблонов в SKILL.md — отвергнута: расхождения неизбежны.
- **Изоляция сабагентов через промпт + slash-command discipline, не через tool-level path whitelist.** Claude Code не предоставляет path-scoped Read для сабагента. Поэтому запрет `arch-reviewer` читать `spec-review.md` — prompt-load-bearing. Митигировано повторением запрета в трёх местах (агент, команда, skill). Хрупко в принципе, но в текущей платформе других опций нет.
- **Русские коммит-сообщения, английские системные промпты.** Прагматичный выбор: задание на русском, история коммитов человеко-читаемая. Системные промпты в `.claude/agents/` на английском — на нём role-holding получается жёстче, и большая часть Claude Code best practices в публичных источниках на английском.
- **Один проект за раз на репозиторий (`process/CURRENT` single-valued).** Подпапки `process/<slug>/` живут параллельно, но активный — один. Если ревьюер запускает свой прогон на другом полигоне, он создаёт `process/<his-slug>/`, и `CURRENT` переключается. Сосуществование артефактов разных запусков обеспечено; параллельное выполнение — нет (это сознательное упрощение).
- **`/audit-code` принимает пути через `$ARGUMENTS`, не извлекает их сам.** Альтернатива — автоматически взять git diff против main — отвергнута как hidden magic. Явное лучше неявного, разработчик контролирует scope.
- **HITL установка `stage: verdicts-applied` — ручная, не команда.** Альтернатива — `/apply-verdicts` команда, которая принимает список accepted/rejected — отвергнута. Применение вердиктов это **редактирование `spec.md`** разработчиком, и команда не может за него это сделать честно. Заставляем человека руками потрогать `STATE.md` — это сигнал «я прочёл, я подумал».
- **Проверка `block`/`major` objections — на стороне `architect`, не на стороне команды.** Команда `/architect` не парсит `spec-review.md` и не валидирует, что каждое блокирующее возражение адресовано в каком-нибудь ADR. Это делает сабагент. Альтернатива — добавить grep-валидацию в команду — была бы безопаснее, но дублирует проверку и фрагилит при изменении формата objection. Пока живём с тем, что валидацию делает агент, который читал и спеку и спек-ревью.
- **Шаблоны имеют `<placeholder>` маркеры, не Jinja-стиль `{{var}}`.** Slash-команды делают plain string-replace при копировании шаблона. Это упрощает дебаг (можно открыть шаблон и руками найти placeholder), но сужает выразительность. Для текущего набора шаблонов хватает.

## Что бы сделали при бесконечном времени

Backlog отсортирован по предполагаемой ценности для оценки задания. Первые два пункта — прямой бонусный балл по ТЗ (метрики наблюдаемости пайплайна и регрессионные тесты на промпты).

- **Метрики через хуки.** Считать сколько objections сгенерировал `spec-skeptic`, какой процент ушёл в `shallow`, сколько было accepted разработчиком, сколько времени стоит каждая стадия. PostToolUse-хук пишет JSONL в `process/<slug>/.metrics`, отдельный slash `/metrics` агрегирует. Это бонусный балл по заданию (наблюдаемость самого пайплайна) и реальная польза для тюнинга промптов.
- **`spec-skeptic` разбить на двух сабагентов (Pass 1 и Pass 2 отдельно).** Сейчас одна модель сама себя оценивает в пределах одного вызова — есть риск self-bias. Две независимые роли (`generator` и `pruner`) с прокидыванием артефакта между ними должны быть жёстче.
- **Регрессионные тесты на промпты.** Фикстура с уклончивыми / противоречивыми ответами разработчика, прогон `/interview` неинтерактивно (через стаб), assert: спека заканчивается с ≥ N `[ASSUMED]` строк, противоречия — с цитированием обеих. Сейчас это ручное упражнение.
- **Path-whitelisted Read для сабагентов.** Если бы платформа давала, мы могли бы hard-isolate `arch-reviewer` от `spec-review.md` — снять prompt-load-bearing запрет. Это запрос фичи к Claude Code, не к нам.
- **Бонусная имплементация одного retry-handler’а** + прогон `/audit-code` на нём — продемонстрировать пост-ревью на реальном коде. Кандидат на Phase 7.
- **Helper для `/audit-code`,** который опционально подставляет git-diff против main как дефолтный `$ARGUMENTS`. Обвёртка вокруг `git diff --name-only`, без магии — чтобы можно было выключить.
- **Multi-language support.** Системные промпты сейчас английские, артефакты разработчик пишет на любом языке. Параметризация языка ответов агента — отдельная развилка.
- **`/status --json`** для machine-readable выхлопа, чтобы можно было обвязать пайплайн внешними инструментами (CI gating, дашборды).
- **Stop-hook на завершение каждой стадии** с автоматической проверкой структуры артефакта (например, после `/challenge-spec` грепнуть, что в `spec-review.md` есть ≥7 пронумерованных objections и таблица Pass 2). Сейчас структуру проверяет только следующий по pipeline сабагент — если он молчит, дефект остаётся в артефакте.
- **`/resume` команда,** которая по `STATE.md` сама определяет следующий шаг и предлагает соответствующую команду. Сейчас это делает `/status` информативно, но требует от разработчика прочитать `Pending human action` и набрать команду руками.
- **Фикстура «hostile developer».** Не просто уклончивый, а активно противоречащий себе и пытающийся уговорить интервьюера согласиться. Цель — увидеть, держит ли модель роль под давлением.

## Компромиссы, которые мы сознательно приняли

- **Бэкенд не пишется** (по заданию). Опциональный маленький кусок — кандидат на Phase 7.
- **`decisions.md` частично заполнен заранее.** Секции «не сработало» и «surprises» появляются только после реального прогона на полигоне (Phase 6). Trade-offs и «при бесконечном времени» можно написать сейчас — они отражают принятые решения по конфигурации, не зависящие от прогона.
- **Не делаем UI / web-обёртки.** По заданию.
- **Не покрываем все мыслимые этапы разработки.** Пять, которые работают, > десяти заявленных.
- **Один общий `process/CURRENT`, без поддержки нескольких параллельных активных проектов.** Параллельные проекты теоретически возможны (отдельные подпапки), но переключение между ними ручное (перезаписать `process/CURRENT`). Multi-active поддержка усложнила бы хук и slash-команды; не оправдано для test-task scope.
- **Нет автоматической транзакционности при падении сабагента.** Если сабагент написал часть `spec.md`, но не успел обновить `STATE.md`, состояние остаётся в полупозиции. Восстановление — ручное (через `/status` + редактирование). Атомарность через временные файлы и rename — кандидат на инфраструктурный апгрейд, но в текущем scope экономически неоправдан.
