# Пайплайн агентов Claude Code для проектирования бэкенд-решений

## 1. Контекст и цель

Тестовое задание (`test-task.md`): построить **процесс проектирования бэкенда** на примитивах Claude Code. Сам бэкенд писать не нужно (за исключением опционального маленького куска для бонусного прогона пост-ревью). Артефакт оценки — содержимое `.claude/`, `CLAUDE.md`, `README.md`, рефлексия и пример реального прогона пайплайна.

Идея процесса: разработчик приходит с задачей, агенты опрашивают его по требованиям, строят спецификацию, оспаривают её, принимают архитектурные решения, проводят ревью результата. На каждом стыке — явный артефакт и явный критерий «готово».

**Полигон для example run:** HTTP-клиент к LLM-провайдеру DeepSeek (retry, rate limiting, streaming, классификация ошибок, таймауты, наблюдаемость). Выбран потому, что плотно набит архитектурными развилками и хорошо нагружает интервьюера и челленджера.

**Не цели:**
- Писать сам бэкенд (кроме опционального куска для пост-ревью).
- Изобретать систему агентов поверх Claude Code — только штатные примитивы.
- Делать UI/обвязку поверх `claude` CLI.
- Покрывать «все мыслимые этапы разработки» — лучше пять рабочих, чем десять заявленных.

## 2. Этапы пайплайна

Пять обязательных этапов из задания + один опциональный шестой между этапами 4 и 5 — для демонстрации post-review на реальном коде:

1. **Опрос разработчика** → `spec.md`
2. **Адверсариальная проверка спеки** → `spec-review.md`
3. **Архитектурное решение** → `adr/NNN-*.md`
4. **Архитектурное ревью** (отдельная роль) → `arch-review.md`
5. **Реализация (опц.)** → `src/* + tests/*` (опционально для демонстрации post-review; роль `implementer` следует утверждённым ADR, пишет узкий чанк под `/implement <scope>`)
6. **Пост-ревью реализации** → `post-review.md` (роль и промпт обязательно; реальный прогон — бонусный)

## 3. Маппинг ролей на примитивы Claude Code

Главное дизайн-решение из задания — какой примитив берётся под какую роль и почему.

| Роль / функция | Примитив | Обоснование |
|---|---|---|
| Интервьюер, Челленджер спеки, Архитектор, Ревьюер архитектуры, Code Auditor (5 ролей) | **Subagents** | Изоляция контекста — главное оружие против sycophancy: челленджер не видит ход мыслей архитектора, видит только итоговый артефакт. У каждой роли свой system prompt и узкий tool whitelist. |
| Implementer (опциональный 6-й сабагент) | **Subagent** | Под бонусную демонстрацию post-review: разработчик-агент, который реализует один узкий чанк (один компонент / класс / эндпоинт) ровно по утверждённой спеке и ADR. Не дизайнер; ADR-IDs цитируются в комментариях кода. Поднимается командой `/implement <scope>`. |
| Точки входа и переходы между этапами: `/start`, `/interview`, `/challenge-spec`, `/architect`, `/review-arch`, `/implement` (опц.), `/audit-code`, `/status` | **Slash commands** | Делают процесс детерминированным и явным для разработчика. Видно, на каком этапе мы и что вызывать дальше. Slash command — тонкая обёртка, которая читает `STATE.md`, выбирает нужный subagent, передаёт артефакты. |
| Шаблоны артефактов (spec, ADR, review report), правила «как писать ADR» | **Skills** | Загружаются по триггеру, не раздувают контекст по умолчанию. Один skill на формат — переиспользуется и интервьюером, и архитектором. |
| Валидация переходов («нельзя в архитектора, пока спека не approved»), сбор метрик пайплайна | **Hooks** | PreToolUse / Stop hooks читают `process/<slug>/STATE.md` и блокируют невалидные переходы. Метрики (число возражений челленджера, сколько принято) — бонусные баллы из задания. |

Сознательно НЕ «всё subagents» — каждый примитив на своём уровне ответственности.

## 4. Структура репозитория

```
swarm/
├── CLAUDE.md                       # точки входа, правила процесса
├── README.md                       # роли, порядок, артефакты, открытые решения
├── test-task.md                    # исходное задание (для контекста)
├── .claude/
│   ├── agents/                     # 5 subagents
│   │   ├── interviewer.md
│   │   ├── spec-skeptic.md
│   │   ├── architect.md
│   │   ├── arch-reviewer.md
│   │   └── code-auditor.md
│   ├── commands/                   # slash commands
│   │   ├── start.md
│   │   ├── interview.md
│   │   ├── challenge-spec.md
│   │   ├── architect.md
│   │   ├── review-arch.md
│   │   ├── audit-code.md
│   │   └── status.md
│   ├── skills/                     # шаблоны артефактов
│   │   ├── spec-template/
│   │   ├── adr-template/
│   │   └── review-rubric/
│   ├── hooks/                      # state-guard, metrics
│   └── settings.json
├── docs/
│   ├── decisions.md                # рефлексия 1 страница (для проверяющих)
│   ├── templates/                  # markdown-болванки артефактов
│   └── superpowers/specs/          # дизайн-спеки (этот документ)
└── process/                        # рабочие папки прогонов
    ├── CURRENT                     # одна строка: slug активного проекта
    ├── deepseek-client/            # наш example run
    │   ├── STATE.md
    │   ├── spec.md
    │   ├── spec-review.md
    │   ├── arch-review.md
    │   ├── post-review.md
    │   └── adr/
    │       ├── 001-retry.md
    │       └── 002-streaming.md
    └── <future-task>/              # папка проверяющего, когда он запустит свой прогон
```

**Принципы размещения:**
- `process/<slug>/` — всё, что произвёл пайплайн на конкретной задаче. Один прогон = одна папка.
- `process/CURRENT` — указатель на активный slug. Все команды работают с `process/$(cat process/CURRENT)/`.
- `docs/` — мета о пайплайне (рефлексия, шаблоны, дизайн-документы), не артефакты прогонов.
- Наш `process/deepseek-client/` — готовый «example run» прямо в репо. Проверяющий запускает `/start <his-slug>`, получает свою папку рядом, не трогая нашу.
- Нумерация ADR — внутри папки задачи (`adr/001-*`, `adr/002-*`), потому что внутри одного проекта ADR-ы реально последовательны.

## 5. Обмен артефактами и resumability

**Всё через файлы в `process/<slug>/`, не через основной контекст.**

`STATE.md` каждого проекта — single source of truth о позиции в пайплайне:
- текущий этап,
- чек-лист выходных условий этого этапа,
- ссылки на уже произведённые артефакты,
- кто следующий (роль/команда).

**Сценарий «разработчик закрыл сессию»:**
1. Открыть новую сессию `claude` в той же папке.
2. `/status` читает `process/$(cat CURRENT)/STATE.md` и восстанавливает картину.
3. `/<next-step>` запускает нужного subagent, передавая ему пути к нужным артефактам.

Subagent получает артефакты через **чтение файлов**, а не из контекста основного агента — это и обеспечивает изоляцию (см. антисикофантию).

## 6. Антисикофантия — конкретные приёмы

«Просто попросил быть критичным» по заданию не считается. Применяем стек приёмов:

1. **Изоляция контекстов через subagents.** Челленджер и ревьюер архитектуры запускаются в своих контекстах и видят только итоговый артефакт (через чтение файла), не диалог его автора. Это убирает социальное давление и якоря из чужой цепочки рассуждений.

2. **Квота возражений.** Челленджер спеки и ревьюер архитектуры обязаны выдать ≥ N (5–7) пронумерованных конкретных проблем с severity, *прежде* чем имеют право на любой вердикт. Шаблон ответа имеет N слотов — нельзя оставить пустыми.

3. **Red-team фрейминг в системных промптах.** Не «оцени качество», а «твоя работа: найти, почему это упадёт в проде в 3 ночи под нагрузкой / при сетевой деградации / при злом пользователе / при miss конфигурации».

4. **Disagree-флаг у ревьюера.** В шаблоне `arch-review.md` обязательное поле «с каким решением архитектора я не согласен». Допускается ответ «ни с каким», но только с обоснованием, почему рассматривал и отверг возражения.

5. **Steelman + worst-case в шаблонах ADR и review.** Заставляет автора пройти по альтернативам и худшему сценарию, прежде чем подтвердить выбор.

6. **Двойной проход у челленджера.** Первая итерация генерирует возражения. Вторая — отдельным запуском того же subagent с другим промптом — оценивает их по рубрике (поверхностное / глубокое / ложное), поверхностные отбраковываются. Это самопроверка против «выдал N штук, лишь бы закрыть квоту».

7. **Метрики через hooks (бонус).** Stop hook считает: сколько возражений сгенерировал челленджер, сколько архитектор принял в ADR, сколько отверг. Низкие числа = сигнал, что роль не работает.

## 7. Human-in-the-loop точки

Жёсткие гейты, без апрува человека пайплайн не двигается:

- **После `spec.md`** — разработчик апрувит или отправляет на доработку.
- **После `spec-review.md`** — разработчик решает, какие возражения принять, какие отклонить. Вердикт фиксируется в `STATE.md`.
- **После каждого ADR** — разработчик выбирает один из вариантов (если архитектор оставил развилку) или подтверждает рекомендацию.
- **После `arch-review.md`** — go / iterate / kill.

Внутри интервью человек естественно в цикле — он отвечает на вопросы. Между вопросами разработчик не апрувит ничего.

Гейты реализованы через hook: `PreToolUse` хук на старте `/<next-step>` команды читает `STATE.md` и блокирует переход, если предыдущий этап не помечен `human-approved`.

## 8. Критерии завершения этапов

У каждого этапа в `STATE.md` — явный чек-лист, привязанный к артефакту:

| Этап | «Готово» = |
|---|---|
| Спека | Заполнены все секции NFR (latency, throughput, отказы/availability, SLA, наблюдаемость, безопасность); нет «TBD»; разработчик написал `approve` (фиксируется в `STATE.md`). |
| Spec review | ≥ N пронумерованных возражений; у каждого статус (`accepted` / `rejected` / `deferred`) с причиной от разработчика; spec обновлена под accepted. |
| ADR | Каждый ADR содержит ≥ 2 рассмотренных альтернатив, явные trade-offs, выбранный вариант с обоснованием, обратные ссылки на пункты спеки. |
| Arch review | Явный вердикт (`block` / `approve-with-changes` / `approve`) + аргументы; обязательно заполненный disagree-флаг. |
| Post-review | Чек-лист «каждое требование спеки и каждое решение ADR — соблюдено / нарушено / не проверяется» с указанием кода. |

Хуки могут проверять часть условий механически — например, наличие N пронумерованных пунктов в `spec-review.md`, наличие секции «Альтернативы» в каждом ADR.

## 9. Поведение при «уклончивых» ответах разработчика

Тестировщики из задания будут смотреть, что делает пайплайн, если разработчик отвечает уклончиво или противоречиво. Поведение интервьюера:

- На противоречие с предыдущим ответом — явное возвращение: «Раньше вы сказали X, сейчас Y, как разрешить?»
- На уклончивое «как лучше» — отказ от роли «дай готовый ответ», возвращение вопроса с конкретными вариантами.
- Лимит: после M (≈3) попыток зафиксировать ответ — пометка `[ASSUMED]` в спеке + явный вопрос разработчику в конце интервью «зафиксировать или переспросить».
- Стоп-условие: интервьюер обязан остановиться, когда чек-лист NFR заполнен, даже если есть желание спросить ещё.

## 10. Воспроизводимость

- `git clone` + `claude` в папке — должно работать без скрытых зависимостей.
- Все требования к окружению (модели, переменные окружения) — в `CLAUDE.md` и `README.md`.
- Стартовая команда (`/start <slug>`) — в `CLAUDE.md` как явная точка входа.

## 11. Открытые вопросы (deferred)

Сознательно не зашиваем в этот спек, фиксируем как решения уровня плана реализации:

- Точные имена и содержимое system prompt для каждого subagent.
- Конкретные tool whitelists для каждого subagent.
- Точная схема `STATE.md` (yaml header? markdown-чек-лист?).
- Точные скрипты hooks (язык: bash / python).
- Содержимое `skills/` (markdown-шаблоны + правила применения).
- Метрики бонусного блока — какие именно собираем, куда складываем.
- Бонус: реальная имплементация маленького куска (один retry-handler) для реального прогона `/audit-code`.

## 12. Бюджет и приоритеты

Бюджет 4–8 часов. Приоритет — рабочий пайплайн на пяти этапах, прогнанный на DeepSeek-клиенте. При нехватке времени режем в порядке убывания приоритета:

1. **(must)** 5 subagents + 5 slash commands + `STATE.md` + один полный прогон в `process/deepseek-client/`.
2. **(should)** Skills с шаблонами артефактов; hook на валидацию переходов; HITL-гейты.
3. **(nice)** Метрики через hooks; двойной проход челленджера; реальная имплементация retry-handler с прогоном `/audit-code`.
4. **(bonus)** Тесты промптов на уклончивые ответы разработчика.
