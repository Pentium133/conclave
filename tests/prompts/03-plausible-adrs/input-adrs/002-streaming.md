# ADR-002: Stream chat completions over Server-Sent Events (SSE)

## Status

proposed

## Context

`FR-2` требует поддержки потокового вывода токенов: потребитель получает токены по мере их генерации провайдером, не ждёт полный ответ. DeepSeek API нативно поддерживает SSE-стриминг через флаг `"stream": true` в payload.

- Drives: FR-2.

## Alternatives

### Alternative A: Server-Sent Events (SSE), как у DeepSeek нативно

- **Cost**: Минимальный — провайдер уже отдаёт SSE, нужно только корректно распарсить. Один HTTP-запрос, одно соединение.
- **Complexity**: Низкая. Парсер: читать построчно, обрабатывать `data: <json>` и `data: [DONE]`. Стандартный `bufio.Scanner`.
- **Correctness**: Каждый event — самостоятельный JSON с дельтой токенов. Гарантирован порядок в рамках одного соединения.
- **Operability**: HTTP-native — проходит через любой HTTP-aware прокси / load-balancer. Знакомый протокол для on-call.
- **Verdict**: chosen.

### Alternative B: WebSocket-канал

- **Cost**: Высокий — DeepSeek не поддерживает WebSocket нативно, потребуется side-car / bridge.
- **Complexity**: Двунаправленный канал, lifecycle более сложный (handshake, ping/pong, явное закрытие).
- **Correctness**: Та же гарантия порядка в одном соединении; преимущество — bidirectional, но нам это не нужно.
- **Operability**: Требует WS-aware инфраструктуры; не все промежуточные прокси поддерживают.
- **Verdict**: rejected — переусложнение под задачу, которая решается one-way стримом.

## Decision

Принимаем **Alternative A**: SSE поверх обычного HTTP-запроса с `Accept: text/event-stream` и `"stream": true` в JSON payload.

- Парсер событий — построчный `bufio.Scanner` с увеличенным `MaxScanTokenSize` (4MB) на случай больших delta.
- Каждое событие пробрасывается потребителю через канал `chan StreamChunk`.
- На `data: [DONE]` канал закрывается, горутина завершается.
- На ошибку парсинга — закрываем канал, последним сообщением кладём `StreamChunk{Err: err}`.

## Consequences

### Positive

- Минимальная задержка first-token: потребитель видит токены по мере генерации.
- HTTP-native — никаких изменений в инфраструктуре (LB, ingress).
- Простая реализация и отладка: SSE — текстовый протокол, можно прочитать `curl -N`.

### Negative

- Однонаправленный канал — потребитель не может отменить/прервать генерацию через тот же канал; для отмены нужен `context.Cancel`, который рвёт TCP-соединение.
- Тарификация: даже если потребитель отменил после первого токена, DeepSeek может успеть генерировать дальше до закрытия TCP. Затраты остаются.

## Open questions

- Как именно сигнализировать потребителю, что соединение оборвалось до `[DONE]`? Сейчас план — закрытый канал + последний chunk с `Err`. Достаточно ли этого?
- Какой таймаут на «нет событий N секунд»? (Сейчас полагаемся на TCP-keepalive на уровне ОС.)
