# Architecture review: image-uploader

## Per-ADR review

### ADR-001: Node.js 22 LTS + NestJS 10 на FastifyAdapter

- **Verdict**: challenge
- **Arguments**:
  - Single-threaded event-loop на pod создаёт жёсткое ограничение, не покрытое в ADR численно: спека NFR-THR-1b требует 200 RPS пика и NFR-CAP-1 — 400 RPS на горизонте. ADR в Negative-секции упоминает «50 одновременных загрузок» как baseline, но не выводит конкретное число pod'ов и не сшивает это с per-pod connection-limit Fastify (`maxRequestsPerSocket: 1000` в ADR-005). Без явной модели `pods × concurrent_uploads_per_pod = peak_rps × p99_duration` остаётся footgun по cost / capacity (ось 4-axes).
  - Решение «отключить body-parser через `addContentTypeParser` с `parseAs: 'stream'`» (FR-5, FR-6) корректно, но в ADR не упомянут конфликт со штатной валидацией Fastify `bodyLimit`: при `parseAs: 'stream'` Fastify НЕ применяет `bodyLimit` к телу автоматически — лимит должен enforced'ить сам handler, считая байты. ADR-005 заявляет `bodyLimit: 100 * 1024 * 1024` как достаточный — это ложное ощущение защиты для FR-6 и оси correctness.
  - Зависимость от `nestjs-pino` для propagation `request_id` через AsyncLocalStorage (ось complexity) принята, но в Negative-секции одновременно сказано «запрет на собственные использования ALS сверх логгер-контекста» — это policy, не механизм. ESLint-правило/CODEOWNERS-проверка не упомянута. Любая будущая интеграция (например, метрики, контекст транзакции БД) уронит NFR-OBS-1 контракт `request_id` молча.
- **3am production failure scenarios**:
  - Trigger: дежурный инженер попытался воспроизвести 415-инциденты локально и поставил `ENV=local` поверх prod-конфига при отладке. Chain: ADR-001 не определяет fail-fast гард на `STORAGE_TYPE=local + NODE_ENV=production` (этот гард упомянут только в ADR-002 Negative), startup проходит, prod-pod начинает писать в `/tmp` контейнера, через 30 минут заполняет ephemeral storage, kubelet evict'ит pod, in-flight загрузки рвутся, NFR-AVL-1 бюджет уходит на 5 минут. On-call видит: всплеск 5xx, потом restart_count растёт по всем pod'ам по очереди, kubectl describe показывает `evicted: ephemeral-storage`.
  - Trigger: пятничный релиз обновлений зависимостей `pnpm update`, в составе — минор `nestjs-pino`, который изменил поведение AsyncLocalStorage propagation на `setImmediate`-callback'ах (известный регресс класса, ADR сам это упоминает в Negative). Chain: `request_id` начинает быть `undefined` в части логов после await S3-uploader'а. Внешний SLO-toolchain (NFR-OBS-1 nota о verifiability) перестаёт корректно строить p95 — все «no-id» события сваливаются в одну корзину. On-call видит: dashboard показывает p95 200ms (ложное), реальные 408 timeout'ы не аттрибутированы. Пользователь жалуется через несколько часов.
- **Operational problems**:
  - Нет alert на «событие лога без обязательного поля контракта NFR-OBS-1» — golden-test в CI ловит только разработчика, не runtime-регресс на проде.
  - Нет capacity model: сколько одновременных 100 MB upload'ов выдерживает один pod при `partSize: 8 MiB × queueSize: 4` AWS SDK (ADR-006) — ~32 MiB на запрос parts + grace-prefix 1 MiB (ADR-005) + S3 SDK keep-alive. При 50 conn/pod это ~1.6 GB RSS — близко к стандартному k8s memory-limit; нет ADR-определённого ratio.
  - Startup +600–900ms NestJS DI graph складывается с readiness probe periodSeconds 5 (ADR-009) и preStop sleep 5 — окно от deploy до RPS-готовности pod'а не зафиксировано в operational README; rolling update может задерживаться больше, чем заложено в `terminationGracePeriodSeconds: 930`.
- **Disagree-flag**: I disagree with: tying the runtime choice to "operational driver монорепо на NestJS", which is treated as load-bearing in this ADR's rejection of Go (Alternative B). The ADR concedes Go is "технически чуть-чуть лучше" on every axis that matters for NFR-LAT-2/NFR-AVL-1 (rate-floor stdlib, no AsyncLocalStorage class of bugs, 7x smaller image, 2.5x lower RSS). This is the single most-on-fire-at-3am service in the platform (anonymous public PUT up to 100 MB), and trading correctness margin for "найм-стек / code-review-discipline" is an explicit cost-shifting from runtime-engineering to ops-during-incident. The ADR should either (a) make the operational driver's weight explicit (e.g., "Go would cost N additional FTE-months over 24 months") or (b) re-rank correctness above operational alignment for THIS service even if the rest of the monorepo stays on NestJS.

### ADR-002: Полиморфная Storage-абстракция через интерфейс `BlobStore`

- **Verdict**: challenge
- **Arguments**:
  - `IfNoneMatch: '*'` для S3-conditional-create — критическая зависимость на FR-12d (ось correctness). ADR честно фиксирует в Negative: «AWS S3 с 2024-11, MinIO ≥ RELEASE.2024-09». Спека требует vendor-agnostic (NFR-DEP-1a), а это решение сужает поддерживаемых вендоров. На S3-совместимых провайдерах без `If-None-Match: *` (Yandex Object Storage, Backblaze B2, Ceph RGW, Cloudflare R2 с разной степенью поддержки) defense-in-depth для FR-12d падает на единственный unique-constraint ADR-003 — **в рамках ADR-004 шага 5** проверка If-None-Match всё равно вызывается. Нет ADR-формализованного fallback'а: что делает upload-handler, если S3-провайдер вернул 501/Not Implemented на If-None-Match?
  - `LocalFsBlobStore.put` через `fs.createWriteStream(path, { flags: 'wx' })` действительно даёт conditional create, но не покрыт в ADR кейс **частично записанного файла после краха процесса**: `wx` падает с EEXIST даже на 0-байт «осколочный» файл, оставшийся от предыдущего попытки. ADR-004 gc удалит метаданные, но физический файл в LocalFS не очистится — ось correctness в dev/test ломается тише, чем в prod.
  - Контракт `url(key)` в интерфейсе тривиален (`${storage.public_base}/${key}`), но это игнорирует FR-3a требование «параметры, достаточные для построения полного пути к **любому** файлу, лежащему в этом хранилище». Если оператор когда-нибудь решит положить объекты не в корень бакета, а под префикс `images/` (что нормальная S3-практика), контракт `key === uuid` без префикс-параметра в storage-сущности уронит инвариант FR-3c. ADR не оставляет места для `path_prefix` в `storages.config` на этом уровне.
- **3am production failure scenarios**:
  - Trigger: ops-команда мигрирует с AWS S3 на Yandex Object Storage по cost-причинам, копирует данные через `s3-copy`, переключает `STORAGE_TYPE=s3` config на новый endpoint. Chain: Yandex Object Storage не поддерживает `If-None-Match: *` на момент миграции (или поддерживает частично, отвечает 501). ADR-004 шаг 5 падает с непредвиденным error-class'ом, который ExceptionFilter (ADR-005) маппит на `internal_error` 500. Все upload'ы до момента, пока ADR-003 unique-constraint не сделает первую остановку, проходят без conditional-write. На-call видит: всплеск 500, рост `s3_put_failed` в логах, недоумение откуда — `If-None-Match` упомянут только в одном ADR.
  - Trigger: dev-инженер во время локальной разработки убил процесс по `Ctrl+C` посреди upload'а 50 MB файла. Chain: `LocalFsBlobStore` оставляет частичный файл с UUID-именем; ADR-004 gc-задача (`*/5 * * * *`) не запустится в локальной dev-сессии, потому что dev-cycle короче 15 минут. Следующий тестовый запуск с тем же UUID получает EEXIST из `wx`, маппится на 409 conflict (FR-12d), разработчик 30 минут думает «у меня же чистая БД». Production-сценарий тот же при kill -9 контейнера: `wx` блокирует retry до gc-цикла + ручной очистки FS-volume в ConfigMap'е, что в k8s pod невозможно (ephemeral). Spec явно декларирует stateless, поэтому LocalFS не должна попасть в prod, но ADR-002 Negative'ом этот гард не enforced'ит — только «рекомендован».
- **Operational problems**:
  - Нет alert на «`HealthCheck` для S3 успешен, но `If-None-Match: *` не поддерживается этим провайдером» — об этом узнаём только когда впервые случится UUID-collision, что при 122 битах энтропии — никогда (и поэтому не ловится канареечной нагрузкой).
  - Runbook для случая «orphan-объект в S3 без метаданных в БД» отсутствует — компенсация через ADR-004 gc удаляет метаданные `pending`, но обратный сценарий (S3-объект без БД-записи, оставшийся от Alternative A-подобного flow в каком-то edge-case) требует отдельного reconciliation-скрипта; такого скрипта в ADRs нет.
  - При migration сценарии «миграция между storage-id» (см. ADR-010 объекция 11) нет процедуры backfill: если оператор добавляет новый storage-id, старые `files` остаются с `storage_id=old`, и публичный URL будет браться из old `public_base`. Нет SQL-проверки в operational README, как массово обновить.
- **Disagree-flag**: I disagree with: making `If-None-Match: *` a hard correctness primitive of `BlobStore.put` (`PutOptions.ifNoneMatch`) instead of a separately-named operation (`putIfAbsent`). Today the interface lets a caller pass `ifNoneMatch: false` and silently get last-write-wins semantics with no compile-time signal that FR-12d invariant just dropped. Renaming to `putIfAbsent` and forbidding the legacy unconditional `put` for the upload-handler path would make FR-12d a type-system invariant rather than a runtime config knob. ADR-006 even passes `IfNoneMatch: '*'` directly into `Upload`, bypassing the abstraction's option, which is itself a smell.

### ADR-003: PostgreSQL 15 + двухтабличная схема `storages` / `files` с FK

- **Verdict**: challenge
- **Arguments**:
  - `storages.config JSONB` без CHECK-валидации (ADR прямо это признаёт в Negative) — **stratup-валидатор** упомянут в одну строку, но сам ADR не определяет схему этого валидатора. Для S3-config полей `endpoint, bucket, region, credentials_ref` нет JSON-Schema; ось correctness держится на дисциплине разработчика. Если оператор по ошибке вставит в `storages.config` `{"bucket": "wrong-name", "endpoint": "..."}`, startup-валидатор должен поймать ДО readiness, иначе все upload'ы пойдут в неправильный бакет с 5xx.
  - `UUID PRIMARY KEY` с random-distribution даёт b-tree bloat и WAL-amplification; ADR упоминает это в Negative «при росте к 400 RPS надо мониторить bloat», но не закладывает ни UUIDv7 как альтернативу (sequential-friendly, поддерживает FR-7 семантику UUID-style, спека НЕ требует именно v4 для server-generated, только NFR-SEC-1a требует CSPRNG — UUIDv7 с random-tail удовлетворяет), ни `BIGSERIAL` + UUID-вторичный-индекс. На горизонте NFR-CAP-1 (200 млн записей) это влияет на ось cost (storage + autovacuum I/O).
  - Поле `bytes BIGINT NOT NULL CHECK (bytes >= 0 AND bytes <= 104857600)` фиксирует FR-6 на уровне БД, но в ADR-004 алгоритм coммита заполняет `bytes` на шаге 4 (INSERT pending) — **до** реального стрима тела. На этом этапе финальный размер ещё не известен. Либо ADR-004 заполняет `bytes=0` и UPDATE'ит на финале (тогда CHECK с `>= 0` бесполезен), либо UPDATE на шаге 6 включает `bytes=detected_bytes` и тогда CHECK имеет смысл, но ADR-004 шаг 6 этого UPDATE не показывает. Это противоречие между ADR-003 и ADR-004 на оси correctness.
- **3am production failure scenarios**:
  - Trigger: оператор добавляет новый storage-id для DR-региона через ручной SQL `INSERT INTO storages (...)`, но в `config` JSONB указывает `endpoint` от старого региона из-за copy-paste. Chain: startup-валидатор не запускается на runtime-INSERT (он только при boot). Все upload'ы, попавшие на этот storage_id, идут в неправильный бакет; healthCheck проходит (HeadBucket на «не тот» bucket — оператор копировал из живого старого storage'а), но S3-multipart upload'ы в нём падают с 403 AccessDenied (старый кредентиал не имеет доступа к новому бакету) или, хуже, успешно записываются в **старый** бакет, и тогда орфаны и метаданные «не сходятся». On-call видит: рост `s3_put_failed`, но `S3HealthIndicator.headBucket` зелёный — конфигурация storage не реcheck'ается на runtime после INSERT.
  - Trigger: рост таблицы `files` достиг ~150 млн строк за 18 месяцев (NFR-CAP-1 пропорционально). Chain: PostgreSQL autovacuum не успевает за UUID-bloat'ом, b-tree fragmentation растёт, INSERT начинает занимать 50–100 ms p95 вместо 5 ms. ADR-004 шаг 4 с `INSERT ... ON CONFLICT DO NOTHING RETURNING id` уносит p95 NFR-LAT-1a (2 секунды) в красную зону. Партиционирование не заложено (ADR Open question прямо это говорит). On-call видит: рост `duration_ms` p95 без явного отказа, медленное выгорание budget'а.
- **Operational problems**:
  - Runbook «как добавить новый storage-id» не существует — ADR-002/003 предполагают что-то-вроде-startup-валидатора, но процедура runtime-добавления не описана; нет separate admin-CLI или migration-tool в ADRs.
  - Нет alert на «доля `INSERT files duration_ms > 50ms` превышает 1%» — это критический leading-indicator для capacity-планирования NFR-CAP-1, и через JSON-логи NFR-OBS-1 на сервере его построить нельзя (`duration_ms` лога — это total request, не per-DB-call).
  - Партиционирование по `created_at` отложено в Open Q — миграция таблицы `files` на партиционирование при 200 млн строк требует zero-downtime инструментов (`pg_partman`, `pg_squeeze`) и downtime-окна; не покрыто в любом ADR.
- **Disagree-flag**: I disagree with: leaving partitioning as an "архитектурного ревью" Open Question instead of a load-bearing decision in this ADR. The same ADR establishes 50 TB / 200M-rows / 24-month horizon as binding (NFR-CAP-1) and acknowledges UUID b-tree bloat. Postponing the decision means at month-12 the team will face an emergency partitioning project mid-flight; declaring monthly RANGE partitioning by `created_at` from day-1 (with a single empty `default` partition for safety) costs nearly nothing now and removes a forced 3am migration later. The ADR should not punt on this.

### ADR-004: INSERT-first → PutObject → UPDATE-finalize; компенсация при отказе

- **Verdict**: challenge
- **Arguments**:
  - Алгоритм шагов 3–7 содержит race с самим собой: на шаге 4 INSERT pending происходит **после** того как pipeline в шаге 3 уже запустился (тело идёт через MinRateTransform + magic-byte peek + S3 multipart Upload). Если шаг 4 возвращает «UUID занят» (CONFLICT), pipeline уже мог отправить первую part'у в S3 (multipart upload создан, parts загружаются). Вызов `upload.abort()` теперь ловит non-trivial state. ADR заявляет «abort multipart, return 409» — но порядок «INSERT → start multipart» был бы безопаснее, чем «start multipart → INSERT». Шаги в Decision-блоке (3, потом 4) противоречат логике «INSERT-first» названия ADR.
  - GC-процесс через `@nestjs/schedule` на каждом pod'е с `SELECT ... FOR UPDATE SKIP LOCKED` — корректен per-row, но при N pod'ах все они опрашивают БД каждые 5 минут. При 4 pod'ах = 4 запроса/5мин по таблице, при 16 pod'ах в peak — 16 запросов. Нет leader-election, нет distributed-lock — потенциально каждый pod держит свою транзакцию, тащит row-lock'и; на 200 млн записей `SELECT ... WHERE status='pending' AND created_at < ...` без partial-index на `(status) WHERE status='pending'` сделает full scan. ADR-003 такого индекса не задаёт.
  - Open question Q3 в самом ADR описывает реальную дыру: «процесс умер ровно после S3 complete, но до UPDATE». Это сценарий **успешный для клиента**, который НЕ получил ответ (TCP reset → retry). gc через 15 минут удалит S3-объект и `pending`-запись. Клиент при retry с тем же `X-Image-Id` получит 200 OK на новой записи и уже-другом теле. Если клиент показывает оба в UI («два моих файла»), это нарушение интуиции. Спека NFR-DUR-1 это допускает; но ADR не маркирует это как **incident scenario** и не пишет runbook.
- **3am production failure scenarios**:
  - Trigger: PostgreSQL primary failover (managed RDS) длится 30 секунд. Chain: 50 in-flight upload'ов на шаге 6 (UPDATE `status='committed'`) получают connection error. ADR-005 `AbortController` пробрасывается, ADR-006 `upload.abort()` вызывается, но S3 multipart уже complete — объект УЖЕ виден в бакете. ADR-004 catch-блок на шаге 6 `0 rows updated → DELETE из S3 (best-effort), return 500` — но `DELETE из S3` тоже потребует connection (S3 может быть здоров, ОК). После failover'а БД восстанавливается. gc через 5 минут найдёт `status='pending'` и попытается `BlobStore.delete()` — но ADR-006 multipart `Upload.done()` уже прошёл, объект существует — `delete` может пройти или упасть. Результат: для клиентов 50 ответов 500; половина из них оставила orphan'ы в БД, четверть оставила orphan'ы в S3, четверть чистые. On-call видит: всплеск `db_insert_failed` (по факту — UPDATE_failed, но `error_class` enum в ADR-008 такого варианта НЕТ — путаница в логах).
  - Trigger: gc-процесс на одном pod'е завис в `BlobStore.delete()` на 90 секунд (S3 timeout не настроен в `BlobStore.delete`). Chain: следующий gc-cycle через 5 минут на том же pod'е стартует, видит ту же запись `pending`, пробует delete — race: оба delete'а на одной и той же S3-key. Один проходит, второй получает 404 (или successful, S3 idempotent). `DELETE FROM files WHERE id=$ AND status='pending'` срабатывает на одном из них; второй DELETE найдёт 0 rows (idempotent). На-call видит: gc-логи запутались, метрики gc-throughput нет (NFR-OBS-1 запрещает счётчики), оператор может только построчно парсить логи.
- **Operational problems**:
  - Нет alert на «доля `status='pending'` старше 30 минут > 100» — leading indicator gc-failure'а; единственный канал NFR-OBS-1 не покрывает gc-event'ы (контракт TerminalEvent в ADR-008 — только для upload-handler'а).
  - Compensation `BlobStore.delete()` в шаге 6 (`0 rows updated`) идёт после S3 multipart уже complete'нут — это полноценный DELETE на public-доступном объекте; если CDN кэширует (NFR-DEP-1c CDN опционален, ADR-010), у клиента может остаться cached 200 OK с public URL, и удаление лагает за CDN purge. В operational README runbook'а на CDN-cache-purge нет.
  - При drain shutdown (NFR-DEP-3, ADR-009) gc-задача останавливается через `OnApplicationShutdown` — это корректно, но если SIGTERM пришёл во время выполнения gc-batch'а, текущий batch может оставить partial state (DELETE из S3 прошёл, DELETE FROM files не успел). Behavior не описан в ADR-004.
- **Disagree-flag**: I disagree with: the ordering "start S3 multipart → INSERT pending" implied by Decision step 3 running before step 4. Naming the ADR "INSERT-first" while the actual control flow opens the multipart upload first creates a window where on UUID collision (step 4 CONFLICT) we already have an in-flight S3 multipart that must be aborted. The safer order is: step 4 (INSERT pending) MUST complete before step 3 (start pipeline). This costs one extra DB roundtrip (~5 ms) before body acceptance and removes the abort-mid-multipart compensation path entirely.

### ADR-005: HTTP-тайм-ауты и slowloris-защита через Fastify config + кастомный `MinRateTransform`

- **Verdict**: challenge
- **Arguments**:
  - `setInterval(...).unref()` per-request — на пике 200 RPS с p99 длительности 5 секунд (NFR-LAT-1a) это ~1000 одновременных таймеров. Каждый тик = 1 итерация по 30-element ring-buffer + reduce. На 1000 таймеров — это 1000 callback'ов/сек, sub-ms каждый, итого ~1ms event-loop'а на тики плюс GC-нагрузка от reduce'а на каждом тике. ADR оценивает «незначимо» без бенчмарка. Open question Q2 признаёт оптимизацию глобальным sweep'ом, но не делает её обязательной — на оси cost / operability при NFR-CAP-1 (400 RPS) это нагрузка, которая ляжет в 10–20% event-loop'а.
  - Спецификация `MinRateTransform` имеет race между `_transform` (вызывается из stream pipeline) и `tick` (из setInterval) — оба пишут в `this.window[idx]` и `this.totalBytes`. Node single-threaded, поэтому на JS-уровне race нет, но **семантика окна** ломается: tick может посчитать сумму до того, как chunk был добавлен в текущую корзину, и vice versa. ADR упоминает «нужны юнит-тесты на race», но сам алгоритм не детерминирован против пограничных таймингов.
  - Wall-clock 900 секунд (NFR-LAT-2c) при 1 Mbps минимума и 100 MB файлов даёт окно 100MB×8 / 1Mbps = 800 секунд + 100 секунд на запись в S3/БД. ADR Negative честно признаёт «легитимный клиент со скоростью 0.95 Mbps будет порезан в 99% случаев». Спека NFR-LAT-2b нижняя граница 1 Mbps — спека сама требует это, ADR не должен «оправдывать»; но на оси correctness ADR забывает фиксировать, что NFR-LAT-1b (10–100 MB best-effort) и NFR-LAT-2c (900s wall-clock) сшиваются впритык — любая retry-логика клиента, не понимающего 408, начинает новый upload, плодя 408-серии.
- **3am production failure scenarios**:
  - Trigger: legitimate mobile-клиент в плохой сети, скорость колеблется 0.7–1.5 Mbps. Chain: 30-секундное окно `MinRateTransform` усредняет — пока средняя в окне ≥ 1 Mbps, всё ОК; но при падении до 0.7 Mbps на 35 секунд `MinRateTransform.destroy(RequestTimeoutError)` рвёт соединение. Клиент видит 408. Retry запускается клиентом с тем же `X-Image-Id` (если клиент его помнит). ADR-004 step 4 INSERT прошёл на первой попытке → 409 conflict на retry, ИЛИ ADR-004 шаг 6 `0 rows updated` запустил DELETE pending, и retry-клиент получит 200 OK. On-call видит: всплеск `error_class='timeout'` коррелирует с регионами с плохой связностью (этого корреляции нет в логах — `client_ip` опционально, ADR-008 Open Q).
  - Trigger: атакующий открывает 200 одновременных PUT-соединений с rate ровно 1 Mbps + 1 байт. Chain: `MinRateTransform` пропускает (выше 1 Mbps), `connectionTimeout` не срабатывает (нет idle ≥ 30s), wall-clock 900s — все 200 сессий висят 900 секунд. Это не slowloris, это «slow-but-just-fast-enough» tarpit. На пике 200 RPS целевой нагрузки + 200 атакующих = 400 одновременных connection — Fastify `maxRequestsPerSocket: 1000` не помогает, это per-socket. Per-pod connection-limit не задан в ADR-005 / ADR-001. На-call видит: `bytes_in` растут, `duration_ms` стабильно ~900000 для атакующих сессий, легитимные запросы ждут в очереди / 503 от лимита connection в Fastify (которого нет).
- **Operational problems**:
  - Нет per-pod max-concurrent-uploads ручки — Fastify не имеет встроенного `maxRequestsInFlight`. NFR-AVL-1 защита от слишком высокой concurrency опирается на k8s HPA, а HPA реагирует на CPU/память с задержкой 30–60 секунд.
  - `error_class='timeout'` в логе единый для всех трёх условий (idle / rate / wall-clock) — operator не может различить «slowloris-атака» от «легитимный плохой канал». Контракт NFR-OBS-1 это разрешает (error_class — закрытый набор), но runbook без подкласса будет шумным.
  - 408-метрика по логам строится post-factum через NFR-OBS-1 nota, нет real-time alert'а на rate of 408 → если волна 408 началась, оператор узнаёт через delay log-ingestion (минуты).
- **Disagree-flag**: I disagree with: the per-request `setInterval` design for `MinRateTransform`. At NFR-CAP-1's 400 RPS peak with p99 of 5s, that's 2000 simultaneous timers. The Open question lists "global sweep over `WeakSet`" as optimization to defer; this should be the chosen design from day one — register each transform in a module-level WeakSet, run a single 1-Hz timer that iterates and calls `tick()` on each. Cost: same lines of code; correctness: identical; operability: bounded scheduler load independent of RPS.

### ADR-006: Тело загружается потоково в S3 multipart, без буферизации на pod

- **Verdict**: challenge
- **Arguments**:
  - `partSize: 8 MiB × queueSize: 4` = ~32 MiB working set на запрос **до того, как первая part flush'нется**. ADR оценивает 50 одновременных = ~1.6 GB RSS, но не учитывает: (1) `MagicBytePeekTransform` буфер 32 байта (ничтожно); (2) `MinRateTransform` ring-buffer (ничтожно); (3) **AWS SDK v3 internal request-queue и retry-buffer** — каждая retry попытка может удерживать part-buffer; (4) Node Buffer pool fragmentation. Реальный RSS на 50 conn легко 2–2.5 GB. Pod limit с запасом должен быть 3–4 GB, что не зафиксировано в operational README.
  - `IfNoneMatch: '*'` в `Upload.params` (т.е. в `CompleteMultipartUpload`) — но `CompleteMultipartUpload` AWS S3 поддерживает If-None-Match только в ограниченных API-версиях. ADR Negative последняя строчка признаёт «частично у прочих провайдеров». Это та же проблема, что в ADR-002 — но здесь критичнее: **multipart-complete** с conditional create — это узкая фича; на MinIO ≥ 2024-09 работает, но Yandex/Backblaze — могут не работать. ADR не определяет fallback (например, HEAD перед Complete с известной race-window).
  - `MagicBytePeekTransform` буферизует 32 байта внутри stream pipe; AWS SDK v3 `Upload` начинает multipart create AsynchroЗneously **до** того, как peek'нет. Это значит: даже на короткое тело (10 KB) `CreateMultipartUpload` API call идёт сразу. Если magic-byte fail'нется — `upload.abort()` нужно отправить. Стоимость: один лишний AbortMultipartUpload API-call за каждое 415 (FR-8 `invalid_format`). При peer'ах с массовым abuse-load'ом это N×$0.000005/req на S3 cost. ADR упомянул эту проблему («~150 лишних S3-запросов в секунду» при малых файлах, $2/день), но не вынес её как **operational alert** на abuse-rate.
- **3am production failure scenarios**:
  - Trigger: атакующий шлёт 1000 запросов/сек с 33 байтами тела (заведомо > 32 для peek, но < 8 MB partSize) и невалидной magic-byte сигнатурой. Chain: каждый запрос → `CreateMultipartUpload` (S3 API call) → 32 байта peek → magic-byte fail → `upload.abort()` (S3 API call) → 415 ответ. На S3 это 2000 API-calls/сек × $5/1M = $0.01/сек = $864/день только на abort'ы. NFR-AVL-1 не страдает (415 — bizvalid), но cost-axis серьёзно. ADR-006 Negative упоминает $2/день на нормальном размере, но не на adversarial. На-call видит: counter S3 API-calls растёт вне корреляции с RPS upload'ов; биллинг алерт через сутки.
  - Trigger: AWS S3 регион experiencing degradation (известный сценарий, AWS публикует 99.9% SLA на S3). `Upload.done()` retry-логика AWS SDK v3 по дефолту делает 3 retry с exponential backoff. Chain: 50 одновременных upload'ов застревают в retry-loop; при retry SDK удерживает part-buffer'ы → RSS растёт; через 60 секунд `requestTimeout: 900_000` ещё не сработал, но pod close to OOM. AbortController из ADR-005 пробросить срабатывание retry — нет (AbortController через `Upload.abortController` отменит retry-цепочку, но это per-request, а не global). На-call видит: «5xx — error_class=s3_put_failed», но кажется единичными — на самом деле каждое retry умножает память, OOM kill в течение 5–10 минут.
- **Operational problems**:
  - Нет S3 lifecycle-rule `AbortIncompleteMultipartUpload` enforced'енного на уровне Terraform/IaC — упомянут как «operational рекомендация в README». Если оператор забыл включить, все killed-mid-upload parts накапливаются и платятся за хранение неограниченно (S3 не чистит сам).
  - AWS SDK retry-policy не упомянута в ADR — defaults: 3 retry, 100ms initial backoff, exponential. Это может удваивать p95 latency NFR-LAT-1a при transient S3 5xx. Лучше явно ограничить retry в `S3Client` config.
  - При abuse magic-byte fail'ов нет rate-limit на вход (Out of scope «Rate-limit реализуется на уровне инфраструктуры»). Но S3 cost — ответственность сервиса. Алерт на `error_class='magic_byte_rejected'` rate > N/s обязателен; не упомянут.
- **Disagree-flag**: I disagree with: starting `Upload` (which fires `CreateMultipartUpload`) before magic-byte sniff completes. The cleaner topology is: peek 32 bytes from `req.raw` first, then conditionally start the `Upload` only if format is valid. This avoids paying `CreateMultipartUpload` + `AbortMultipartUpload` on every 415, removes the adversarial-cost amplification, and simplifies the error path (no abort-mid-multipart code). The cost is a tiny lookahead on `req.raw` (PassThrough peek) before pipeline kickoff; complexity-axis trades a bit of glue for removing a whole error-state.

### ADR-007: Magic-byte валидация — табличный sniffer первых 32 байт

- **Verdict**: accept
- **Arguments**:
  - Прямое 1:1 отображение FR-11 в `TABLE` массив — ось correctness идеальна для unit-тестирования и code-review. Открытый массив с явными `match`-функциями делает изменение whitelist'а атомарным (одна строка add/remove + один тест).
  - Отказ от `file-type` npm как зависимости (Alternative B) — корректное обоснование на оси cost/security: пакет распознаёт SVG (явно исключённый FR-11), и любая ложноположительная классификация на upgrade пакета может протечь сквозь whitelist в обход спеки.
  - `_transform` callback с `cb(error)` ДО `this.push(chunk)` гарантирует, что magic-byte fail рвёт pipeline до того, как S3 получит первый байт тела (через `pipeline()` propagation) — это формально удовлетворяет FR-10 «файл в storage НЕ сохраняется».
- **3am production failure scenarios**:
  - Trigger: клиент шлёт PNG, но первый TCP сегмент содержит только 7 байт (нестандартный MTU + chunked transfer encoding неожиданным образом). Chain: `_transform` ждёт ≥ 32 байт, получает 7, возвращает `cb()` без push'а. Следующий чанк не приходит ещё 25 секунд (transient network delay). `MinRateTransform` (ADR-005) считает это idle ниже rate-floor через 30 секунд → `RequestTimeoutError`. Клиент видит 408 несмотря на корректный PNG. On-call видит: `error_class='timeout'`, `bytes_in=7`, `detected_format='none'` — путаница: формат не определился, но и валидация не падала.
  - Trigger: `_flush` вызывается на теле 8 байт. Chain: `this.buf.length >= 12` ложно (PNG сигнатура 8 байт, но в `_flush` проверяется `>= 12` для покрытия WebP). PNG сигнатура была валидной, но sniff не запустился — `cb(InvalidFormatError('body too short or unknown format'))`. Клиент с валидным 8-байтным «PNG signature only» (теоретически возможно при тестовых payload'ах) получает 415, хотя сигнатура корректна. Edge-case, но ADR Negative его не упоминает.
- **Operational problems**:
  - Нет alert на «`detected_format='none'` rate > 5%» — leading-indicator либо abuse-волны, либо сломанного клиента после deploy'а.
  - Логирование `detected_format` происходит только на успешном sniff'е; при fail'е через `cb(error)` peek-buffer пропадает — debug-режим (NFR-OBS-1 разрешает debug-уровень) не показывает первые 32 байта запроса для post-mortem'а. Нужно явно логировать `head[0..7]` hex на 415-fail.
- **Disagree-flag**: I considered the following objections and rejected them because:
  - **Objection A**: `MagicBytePeekTransform` should peek without buffering at all — call `req.raw.once('readable')` and `read(32)` synchronously. Rejected because: this couples the magic-byte logic to `req.raw` lifecycle (consume-once, no peek-back), making the pipeline harder to compose with `MinRateTransform` and `Upload.body`. The Transform-stream design is correctly pluggable.
  - **Objection B**: detection threshold 32 bytes is excessive — 12 is sufficient (max minLen). Rejected because: the ADR's reasoning ("запас на случай будущих whitelist-расширений") is a 20-byte cost on the first packet (typically 1.4KB MSS), invisible at the relevant scale, and removes a coupling between FR-11 evolution and the constant.

### ADR-008: Структурный логгинг через `nestjs-pino` + AsyncLocalStorage

- **Verdict**: challenge
- **Arguments**:
  - NFR-OBS-1 явно требует поле `ts` (RFC 3339, UTC, мс). ADR Negative последняя строка признаёт: «Pino пишет в поле `time`, не `ts`. Митигация: настроить `customProps` или transformer для переименования (тривиально через `formatters.log`)». Это **contract violation**, оставленный в Open question статусе — golden-test в CI должен проверять имя поля **ровно `ts`**, а Decision-блок не показывает конкретный конфиг `formatters.log`. Если тривиально — почему не зафиксировано в Decision?
  - `error_class` enum фиксирован в TS-типе, но `db_insert_failed` маппит и INSERT и UPDATE-failures (ADR-004 шаг 6 — это UPDATE, не INSERT). Класс ошибки `db_insert_failed` для UPDATE-ошибки лжёт оператору в логе. Контракт NFR-OBS-1 разрешает «иные коды» с тем же контрактом, но enum в ADR-008 не покрывает `db_update_failed`.
  - Pino sync vs async writer: дефолт async (`{ sync: false }`) буферизует ≤4 KB или ≤10 ms. На SIGKILL pod (out-of-memory, ADR-006 sce­ne, kubelet eviction) последние ≤10 ms логов теряются. ADR-008 Open question Q3 признаёт это и спрашивает «допустимо?» — но при 200 RPS это 2 события в окне 10 ms, и именно в момент SIGKILL критично сохранить терминальные записи о in-flight upload'ах. Должно быть `sync: true` для production по NFR-OBS-1 verifiability.
- **3am production failure scenarios**:
  - Trigger: внешний SLO-toolchain (parsing JSON-логов) обновляется, ожидая поле `ts`; разработчик не успел настроить `formatters.log`. Chain: все логи имеют `time` (Pino default), `ts` отсутствует. SLO dashboard показывает 0% доступности (потому что фильтр `WHERE ts > now()-1h` ничего не находит), pager горит. Реальный сервис здоров. On-call видит: «доступность 0», тратит 30 минут разбираясь, что dashboard сломан, не сервис.
  - Trigger: pod получает SIGKILL после OOM (см. ADR-006 scenario). Chain: 32 in-flight upload'а — для 30 из них S3-multipart прошёл, UPDATE `status='committed'` ещё не успел; pino async-writer держит 28 терминальных записей в буфере. SIGKILL — буфер потерян. Записи `pending` остаются, gc через 15 минут удалит их (включая S3-объекты, см. ADR-004 Open Q3). Клиенты получили connection reset, retry с тем же `X-Image-Id` сделает второй upload. Логи о первой попытке отсутствуют — root-cause analysis в логах невозможен. На-call видит: пустота за 10 ms перед SIGKILL'ом.
- **Operational problems**:
  - Поле `client_ip` (Open Q1) не зафиксировано — без него нельзя отличить abuse-source от legitimate retry-storm; runbook на «inflate of `error_class='timeout'`» бесполезен.
  - `LOG_LEVEL=debug` через ENV override (ADR-008) для UUID файла в логах — нет ADR-определённого rate-limit'а на debug-объём; включение debug-level в проде на 200 RPS = 200 lines/sec extra с UUID и трассой — может уронить log-pipeline.
  - Контракт-тест golden-file существует, но runtime-проверки на каждом deploy'е что Pino не сменил формат после bump'а версии — нет; integration-test «каждый response пишет одну терминальную запись» упомянут в ADR Negative, но не вынесен как обязательный pre-merge gate.
- **Disagree-flag**: I disagree with: leaving `time` → `ts` field rename, sync vs async writer, and `client_ip` inclusion as Open Questions / "тривиально". All three are direct NFR-OBS-1 contract requirements (rename) or production-incident root-cause inputs (sync writer, client_ip for rate-correlation). They should be in Decision-block with concrete `formatters.log` config and `pino.destination(1, { sync: true })` settings, not deferred to "арх. ревью".

### ADR-009: Graceful shutdown — `app.enableShutdownHooks()` + Fastify `app.close()` + `@nestjs/terminus`

- **Verdict**: challenge
- **Arguments**:
  - `terminationGracePeriodSeconds: 930` * 4 pod'а rolling-update по одному = до 1 часа на полный rollout. ADR Negative это признаёт. Для security-hotfix'а это **неприемлемо** — критическая CVE в magic-byte sniffer (FR-10 / FR-11) или в AWS SDK не может ждать час. ADR не определяет escape hatch (fast-rolling с force-kill через ConfigMap флаг, или blue-green как fallback). Ось operability на security-инциденте проиграна.
  - Readiness `db.pingCheck` + `s3.headBucket` каждые 5 секунд = 0.4 RPS на БД и S3 от probe-нагрузки на 4 pod'а. ADR говорит «незначимо». Но **на S3-стороне** `HeadBucket` — это API call ($0.0004/1000 req); 0.2 req/sec × 86400 sec/day × 30 = 518400 req/мес × 4 (пиковая нагрузка) = ~2M req/мес = $0.80/мес. Незначимо в долларах, но при дегрейде S3-региона все 4 readiness-probe начинают флапать одновременно — все 4 pod'а уходят в NotReady **одновременно** → 100% downtime во время transient S3 incident. Liveness отделён, поэтому pod'ы не рестартуют, но новые запросы не маршрутизируются — выгорание NFR-AVL-1 за минуты.
  - `OnApplicationShutdown` хуки выполняются последовательно по dependency-graph (ADR Negative предупреждает). gc-задача (ADR-004) на shutdown — куда её положить в graph? ADR-004 говорит «через `OnApplicationShutdown` lifecycle-хук», но не уточняет, ДО или ПОСЛЕ закрытия DB pool. Если ПОСЛЕ — gc упадёт на DB-disconnected; если ДО — gc может успеть запустить новый batch и не успеть его завершить. Конкретный ordering не зафиксирован.
- **3am production failure scenarios**:
  - Trigger: критическая 0day в `MagicBytePeekTransform` (например, prototype pollution через специальный chunk), pater'у нужен hotfix немедленно. Chain: `kubectl rollout restart` запускает rolling update; первый pod уходит в drain, ждёт in-flight 100 MB upload'ы до 900 секунд. 50 атакующих сессий специально шлют по 1.1 Mbps → проходят `MinRateTransform`, держат соединения 800 секунд. Drain-окно почти полностью используется на ХАК-сессии (легитимные клиенты тоже там). Hotfix занимает 1 час полного rollout'а. На-call видит: rolling-update «в процессе» 60 минут, security incident remained exploitable.
  - Trigger: managed PostgreSQL undergoing maintenance failover (ожидаемая операция, 30 секунд). Chain: `db.pingCheck` падает на всех 4 pod'ах одновременно → все 4 readiness=NotReady → load balancer убирает все pod'ы из ingress → 100% 503 для клиентов в течение 30+ секунд (failure threshold=2 × 5 sec = 10 sec до NotReady, плюс восстановление). NFR-AVL-1 за один failover теряет ~30 sec × 100% downtime = бюджет 43 мин/мес сжигается на 1.2%; за 10 maintenance failover'ов в месяц — больше бюджета. На-call видит: 100% 5xx во время каждого RDS event'а, но liveness=OK → нет рестартов.
- **Operational problems**:
  - Нет «degraded mode» для readiness: при partial S3 outage сервис должен оставаться available для тех клиентов, чей upload не зависит от S3 (нет таких — тут unit) ИЛИ должен возвращать 503 целенаправленно. Текущий бинарный ready/not-ready без middle-ground.
  - Нет alert на «`/healthz/ready` returns 503 более N секунд» — единственный сигнал — внешний blackbox monitoring (не часть NFR-OBS-1).
  - `preStop sleep 5` в YAML фиксирован в operational README, не в коде — рассинхрон между ADR (где «5») и оператором (который может поставить 0 без code-review-gate).
- **Disagree-flag**: I disagree with: making readiness binary on (db AND s3 reachable). For a service whose upload path depends on both being healthy at the moment of request, this is correct *for new requests*; but `failureThreshold: 2 × periodSeconds: 5` means a 30-second transient DB failover blackholes ALL pod traffic for 10s+. The probe should add jitter (random 0–2s pre-check delay per pod) so that 4 pods don't fail probe in lockstep, OR readiness should use a sliding-window of last N pings (e.g., "5 of last 10 succeeded") to absorb transient blips. The current design literally amplifies dependency outage to 100% service outage for the duration.

### ADR-010: Публичный URL — `<storages.public_base>/<files.id>`, без проксирования

- **Verdict**: challenge
- **Arguments**:
  - `Content-Type: image/*` выставляется на S3-метаданном (ADR-007 + ADR-002), но `X-Content-Type-Options: nosniff` и `Content-Disposition: attachment` НЕТ (ADR Negative честно фиксирует). Без `nosniff` браузер может MIME-sniff'ить тело — для polyglot JPEG+HTML это stored-XSS на собственном домене (`public_base`). ADR ссылается на «объекция 3 ревью отклонена», но **отказ доказывает**, что риск принят, не отсутствует. На оси correctness/security это явная лазейка под `domain= images.example.com`.
  - Public-read бакет (Decision: «public-read ACL ИЛИ за CDN с public-read origin policy») — единственное упоминание в operational README. Нет terraform-as-code или admission-controller в k8s, который бы блокировал deploy сервиса при private-read бакете. Если оператор по ошибке настроил bucket private, сервис стартует, healthCheck зелёный (HeadBucket работает с креденшалом), но клиенты получают 200 OK с URL, который возвращает 403 при скачивании.
  - Изменение `storages.public_base` после первого использования — operational README предупреждение, не enforced'ило ничем. Спека FR-9a «URL не имеет TTL — он валиден неограниченно долго». ADR Negative прямо признаёт «если оператор сменит — все ранее выданные URL невалидны». То есть **FR-9a соблюдается только до первого operator-error'а** на оси durability.
- **3am production failure scenarios**:
  - Trigger: оператор настроил CDN перед бакетом, обновил `storages.public_base` с `https://s3-direct.example.com` на `https://cdn.example.com` без save-old-mapping. Chain: все ранее выданные URL клиентам, использующие `https://s3-direct.example.com/<uuid>`, продолжают работать **только** если DNS s3-direct.example.com всё ещё указывает на бакет — но если оператор удалит DNS, URL мертвы. Невозможно отличить «правильный» сценарий от «неправильного» через probe сервиса. На-call видит: customer support tickets «мой URL не работает», логов нет (download out-of-scope, не проходит через сервис).
  - Trigger: атакующий обнаруживает polyglot exploit в популярной image-парсер-библиотеке на стороне жертвы (например, Pillow CVE). Chain: атакующий загружает polyglot JPEG+HTML через анонимный upload (Out of scope «аутентификация»). magic-byte sniffer проходит (FR-11). URL валиден. Атакующий распространяет URL через социальные сети, под доменом `images.example.com`. Жертвы открывают, браузер MIME-sniff'ит → выполнение HTML/JS. Сервис ничего не знает (download out-of-scope). На-call узнаёт через abuse-reports / external incident, не из логов сервиса.
- **Operational problems**:
  - Нет процедуры takedown'а конкретного UUID — ADR-002 даёт `BlobStore.delete(key)`, но он не expose'нут наружу как endpoint. Оператор должен пойти SQL-DELETE + S3-delete вручную; runbook отсутствует.
  - Нет процедуры массового мониторинга «сколько объектов в бакете без записи в БД» — orphan-detection (ADR-004 Open Q3 сценарий gc-удалил-успешный) требует bucket-LIST + JOIN с DB, в production-нагрузке (50 TB) это часы full-scan.
  - При CDN перед бакетом — runbook на cache-purge для удалённого UUID отсутствует; такой gap может оставить «удалённый» файл доступным несколько часов.
- **Disagree-flag**: I disagree with: deciding NOT to set `X-Content-Type-Options: nosniff` and `Content-Disposition: attachment` on S3 object metadata. The ADR cites "objection 3 ревью отклонена разработчиком" as justification, but objection-rejection is a non-technical reason; the cost of adding two more `Metadata` fields in `Upload.params` is zero and the security benefit is non-trivial (defense against polyglot stored-XSS under your own domain). This is a free correctness improvement that the architecture is leaving on the table.

## Cross-cutting issues

- **Observability story в целом — несогласованность контракта vs реализация**: ADR-008 декларирует Pino-output с полем `time`, но контракт NFR-OBS-1 требует `ts`. ADR-008 откладывает это в Negative как «тривиально». Это нарушение контракта на уровне deploy. Конкретный сигнал, которого нет: **golden-file тест в CI, проверяющий exact-field-name `ts`** на каждом merge — не фигурирует в Decision-блоке ADR-008. Без этого через 2 месяца после деплоя при rebase зависимостей контракт молча сломается.
- **Observability story — отсутствие per-stage timings**: `duration_ms` в NFR-OBS-1 — total request. Нет полей `db_insert_ms`, `s3_upload_ms`, `magic_byte_ms`, `db_update_ms`. Когда p95 NFR-LAT-1a (2 секунды) пробивается, оператор не может определить ИСТОЧНИК медленного шага из логов сервиса. Конкретный недостающий сигнал: на терминальной записи лога должны присутствовать 4 stage-timings (ADR-008 этого не закладывает, NFR-OBS-1 не запрещает добавлять).
- **Observability story — UUID файла недоступен для post-mortem**: NFR-SEC-1c запрещает UUID в публичных логах. ADR-008 решает «debug-only». При поддержке (3am инцидент: «у меня файл вернул 5xx, вот мой response с UUID») оператор не может найти запись в логе по UUID — она не индексируется. Конкретный недостающий сигнал: **hash(UUID) в info-логе как fingerprint** позволил бы корреляцию без exposure'а самого UUID.
- **Failure isolation — readiness-probe lockstep**: ADR-009 readiness падает на всех pod'ах одновременно при transient depend-failure. Blast radius: 100% сервиса при 30-секундном RDS failover. Это явно противоречит цели NFR-AVL-1 99.9%. Никто из ADRs не вводит per-pod jitter или sliding-window.
- **Failure isolation — S3 abuse cost amplification**: ADR-006 magic-byte fail происходит ПОСЛЕ создания multipart upload'а. Атакующий с N байт мусора генерирует 2N S3 API calls. Ни один ADR не определяет cost-budget alert или rate-limiter на pre-S3-upload event'ы.
- **Deployment — incompatible с NFR-DEP-2 для security-hotfix**: NFR-DEP-2 требует rolling update; ADR-009 требует `terminationGracePeriodSeconds: 930` (15.5 минут на pod). На 4 pod'ах = 1 час полный rollout. NFR-AVL-1 99.9% даёт 43 минуты/мес; полная rollout длится больше окна допустимой downtime. Нет blue-green / canary fallback'а в любом ADR. Rollback story: rolling-update обратно — те же 1 час; нет emergency-flush механизма.
- **Deployment — отсутствие migration ordering**: ADR-003 определяет схему, ADR-004 определяет алгоритм с `status='pending'`. Но миграции (как именно создать таблицы первого раза, как накатывать ALTER) не покрыты ни в одном ADR. Нет канона — Prisma Migrate, Drizzle Kit, Atlas, raw SQL? ADR-001 Open Q признаёт это не-блокером. Это блокер для первого деплоя.
- **Cross-ADR contradiction — `bytes` колонка**: ADR-003 объявляет `files.bytes BIGINT NOT NULL CHECK (bytes >= 0 AND bytes <= 104857600)`. ADR-004 шаг 4 INSERT pending — на этом этапе `bytes` ещё не известен (тело ещё не считано). Шаг 6 UPDATE `status='committed'` — где UPDATE'ит `bytes`? Не показано. NOT NULL on INSERT pending форсирует значение; единственный вариант `bytes=0` — формально нарушает спирит спеки FR-6. Это не покрыто.
- **Cross-ADR — отсутствие FK ordering для `INSERT … storage_id`**: ADR-003 fixed FK `files.storage_id → storages.id`. ADR-004 шаг 4 INSERT files использует `storage_id`. Если `storages` запись не найдена при boot'е (operator забыл наполнить), сервис должен fail-fast на startup — но ни один ADR не определяет startup-валидатор «active storage записан в БД». Сервис стартует, healthCheck зелёный (S3 reachable), первый upload падает на FK-нарушении → 500 internal_error.

## Final verdict

- **Verdict**: iterate
- **Required follow-ups before next stage**:
  1. **ADR-004 ревизия**: исправить порядок шагов 3–4 — INSERT pending DOLZHEN завершиться ДО открытия S3 multipart upload. Документировать поведение gc на shutdown во время выполнения batch'а. Зафиксировать ordering `OnApplicationShutdown` хуков (gc → close DB → close S3).
  2. **ADR-008 ревизия**: вынести из Open Questions в Decision: (a) `formatters.log` для переименования `time → ts` с golden-test'ом; (b) `pino.destination(1, { sync: true })` для prod; (c) включение `client_ip` (нормализованный из `X-Forwarded-For`); (d) расширение `error_class` enum для `db_update_failed`.
  3. **ADR-009 ревизия**: добавить per-pod jitter в readiness-probe (random 0–2s pre-check delay) ИЛИ sliding-window probe (5 of last 10 succeeded), чтобы 4 pod'а не уходили в NotReady в lockstep при transient dep-failure. Зафиксировать `OnApplicationShutdown` ordering для gc-задачи. Добавить escape-hatch для security-hotfix (fast-rolling флаг с force-close через 60s, или canary-канал).
  4. **ADR-010 ревизия**: добавить `X-Content-Type-Options: nosniff` и `Content-Disposition: attachment` в `Upload.params.Metadata` — нулевая стоимость, защита от polyglot stored-XSS. Документировать procedure takedown'а UUID (S3-DELETE + DB-DELETE + CDN-purge) как формальный runbook в operational README.
  5. **ADR-006 ревизия**: переместить magic-byte peek ДО `CreateMultipartUpload` (peek через PassThrough на `req.raw` синхронно с первым chunk'ом перед запуском pipeline) — устраняет cost-amplification на adversarial 415-нагрузке и упрощает abort-path. Зафиксировать AWS SDK retry-policy (3 retry → 1 retry с 100ms backoff) явно в `S3Client` config. Memory-limit рекомендация 3-4 GB на pod в operational README.
  6. **ADR-005 ревизия**: заменить per-request `setInterval` на module-level WeakSet + global 1-Hz sweeper (как в Open question Q2). Добавить per-pod max-concurrent-uploads семафор как defence-in-depth перед HPA. Зафиксировать `keep-alive` semantic для tarpit-протекции (1 Mbps + 1 byte attack).
  7. **ADR-003 ревизия / новый ADR**: monthly RANGE-partitioning по `created_at` для `files` с дня 1; partial index `(status) WHERE status='pending'` для gc-производительности; уточнить семантику `bytes` колонки (NULLABLE на pending → UPDATE на commit с `NOT NULL` на committed-state — через CHECK constraint на статус-зависимый).
  8. **ADR-002 ревизия**: переименовать `put` с `ifNoneMatch` опцией в отдельный метод `putIfAbsent` для type-system enforcement FR-12d. Определить fallback для S3-провайдеров без поддержки `If-None-Match: *` — головная компонента defense только на DB unique-constraint; зафиксировать список «verified-supported» провайдеров и runtime-probe (попытка `PutObject` с `If-None-Match: *` на startup-валидаторе → fail-fast если 501).
  9. **Новый ADR (миграции БД)**: канонический tool (Prisma Migrate / Drizzle Kit / Atlas / raw SQL), процедура накатки на первом деплое, процедура отката при failed-deploy, ordering между application-deploy и schema-migration в k8s.
  10. **Cross-cutting — capacity model в ADR-001 или новом ADR**: `pods × concurrent_uploads_per_pod × avg_upload_duration = peak_rps`. Зафиксировать N pod'ов для NFR-THR-1b 200 RPS пика и для NFR-CAP-1 400 RPS, memory-limit на pod, expected event-loop utilization. Без этой модели HPA-конфиг — guesswork.
  11. **Cross-cutting — observability расширение**: добавить per-stage timings (`db_pending_ms`, `s3_upload_ms`, `magic_byte_ms`, `db_commit_ms`) в TerminalEvent (ADR-008) для root-cause при NFR-LAT-1a regress. Hash(UUID файла) как fingerprint в info-логе для 3am-correlation без NFR-SEC-1c violation.
