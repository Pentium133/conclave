# Spec: deepseek-client

## Goal

A standalone HTTP proxy/gateway service that sits in front of the DeepSeek LLM API and provides retry, rate limiting, provider-error handling, timeouts, and observability to its callers. Streaming (SSE pass-through) is mentioned as an aspirational capability of the proxy family but is explicitly deferred from this iteration (see §Out of scope, FR-4).

## Functional requirements

- **FR-1**: The proxy holds the upstream DeepSeek API key server-side as part of its own configuration. Callers MUST NOT supply DeepSeek credentials; the proxy injects the upstream `Authorization` header itself when forwarding to DeepSeek.
- **FR-2**: The proxy exposes a single upstream-mirroring endpoint: `POST /v1/chat/completions`. No other DeepSeek endpoints are surfaced in this iteration.
- **FR-3**: The proxy mirrors DeepSeek's `POST /v1/chat/completions` request and response JSON schemas verbatim, without modification, additions, or renaming of fields. Drop-in compatibility goal: a caller already using the DeepSeek SDK / raw HTTP client MUST be able to switch to the proxy by changing only the base URL (and removing the upstream API key from the client side, since FR-1 injects it server-side).
- **FR-4**: This iteration handles non-streaming chat completions only. The proxy MUST accept requests where `"stream"` is `false` or omitted (defaulting to non-streaming). Requests with `"stream": true` MUST be rejected with HTTP 400 and a JSON error body explaining that streaming is not supported in this iteration; the proxy MUST NOT silently downgrade `stream: true` to `stream: false`. Detection of `stream: true` is performed without parsing the full body into a typed structure (see FR-11): the proxy MAY peek at the request body for the literal substring/JSON token `"stream": true` (within size and depth limits compatible with FR-8), or MAY scan only the top-level `stream` field via a streaming JSON tokenizer; either approach is acceptable.
- **FR-5**: Retry policy. Two classes of failure trigger automatic retry:
  - **Class A — transport-level failures (no response bytes received)**: DNS resolution failure, TCP connection error (ECONNREFUSED / ECONNRESET / EHOSTUNREACH / ENETUNREACH), TCP reset before any HTTP response bytes have been received, TLS handshake error, and read timeout BEFORE any response bytes have been received from upstream.
  - **Class B — upstream HTTP 502 and 503 (edge-shaped failures)**: HTTP 502 (Bad Gateway) and HTTP 503 (Service Unavailable) responses from upstream are treated as edge-load-balancer or upstream-availability failures that very likely mean the request did not reach the model; the rationale for not retrying non-idempotent calls bites less here than it does for 500/504. If upstream supplies a `Retry-After` header on the 502/503, the proxy MUST honour it (use it as the backoff delay, capped by the FR-6 budget); otherwise FR-6 backoff applies.
  - **Not retried**: any other HTTP response from upstream — including 4xx, 500 (Internal Server Error: model may have run), 504 (Gateway Timeout: upstream-side timeout, ambiguous), 429 (rate-limited; forwarded as-is per the upstream-429 contract in FR-7), and any 3xx — MUST be forwarded to the caller as-is and MUST NOT be retried.
  - **Cutoff**: "before any response bytes received" applies to Class A. Once upstream begins streaming response bytes, the proxy treats the call as committed and does not retry, regardless of any subsequent transport error mid-body. Rationale: chat completions are non-idempotent on the upstream side (billing, generation state); retrying after upstream may have processed risks double billing and divergent generations.
- **FR-6**: Retry budget (conservative profile). Maximum attempts per logical caller request = **2** (1 initial attempt + at most 1 retry). Backoff between the initial attempt and the retry = **fixed 500 ms with ±20% uniform jitter** (delay drawn uniformly from [400 ms, 600 ms]). When FR-5 Class B retry is triggered with an upstream `Retry-After` header, the backoff is the lesser of (the upstream-supplied `Retry-After` in seconds) and (the wall-time remaining under NFR-LAT-1 minus a one-attempt budget); if the upstream `Retry-After` exceeds remaining wall-time, the proxy MUST NOT retry and forwards the 502/503 to the caller. The retry budget applies only to the failure classes enumerated in FR-5; it does not apply to other upstream HTTP responses.
- **FR-7**: 429 contract — local outbound overflow vs upstream 429.
  - **Local outbound overflow**: when the outbound rate cap (NFR-CAP-1) is exhausted at the moment a caller request would be issued upstream, the proxy MUST respond to the caller immediately with HTTP **429 Too Many Requests**. The response includes a `Retry-After` header set to the integer number of whole seconds (rounded up, minimum 1) until the next outbound token becomes available, and an additional `X-RateLimit-Reset-Ms` header with the millisecond-precision wait until the next available token (so callers that respect the finer header can avoid the rounded-up over-advise). The response body is a JSON error indicating "outbound capacity exhausted". The proxy MUST NOT queue, buffer, or delay the request internally; rejection is immediate. FR-5/FR-6 retry semantics are not engaged for local-overflow 429.
  - **Upstream 429 forwarded**: when DeepSeek returns HTTP 429, the proxy MUST forward the response to the caller as-is, including DeepSeek's own `Retry-After` header verbatim (no rewriting, no replacement, no addition of `X-RateLimit-Reset-Ms`). The `outcome` metric label is `rate_limited` for this case as well, but the upstream-status-code metric (NFR-OBS-1) records the underlying 429.
  - **Both simultaneously true**: if local outbound bucket is empty AND upstream had previously returned 429 — the local-overflow check fires first (the upstream call was never issued), so the local-overflow response wins.
- **FR-8**: Inbound request body size cap. The proxy MUST reject inbound caller requests whose `Content-Length` header exceeds **256 KiB** (262144 bytes) with HTTP **413 Payload Too Large** and a JSON error body, before any upstream call. Requests without a `Content-Length` header (e.g. chunked transfer-encoded) MUST be rejected with HTTP 411 Length Required; the proxy does not buffer streaming inbound bodies. This cap is well above the size of typical chat-completion requests (a 256 KiB JSON can express prompts in the high-tens-of-thousands-of-tokens range) and is a DoS guardrail, not a product limit. The cap value is fixed for this iteration; making it operator-configurable is an architect-stage decision.
- **FR-9**: Single-instance enforcement and listening contract.
  - The proxy MUST listen on a single TCP port for inbound caller traffic (default `8080`, overridable via env `PORT`).
  - The proxy MUST listen on a separate TCP port for the metrics/health endpoint (default `9090`, overridable via env `METRICS_PORT`). See NFR-OBS-1 and Objection-10 reasoning for why metrics are on a separate port.
  - On startup, if either port is already in use, the proxy MUST exit non-zero with an explicit error, NOT silently retry on a different port.
  - This bind-or-die behaviour is the proxy's contribution to "single-instance" enforcement: combined with NFR-DEP-2's deployment procedure (stop-then-start, host-port mapping in Docker), it makes a second concurrent instance fail loudly on its first second of life.
- **FR-10**: Caller correlation header. Every response from the proxy to the caller (success or error, all status codes) MUST include an `X-Request-Id` HTTP header whose value equals the proxy-assigned `request_id` recorded in the per-request log line (see NFR-OBS-1). This lets a caller hitting an error correlate their incident with the proxy's logs without operator manual lookup. Caller-supplied `X-Request-Id` headers on the inbound request are explicitly **NOT** echoed back, NOT used as the proxy's `request_id`, and NOT honoured for any purpose in this iteration; the proxy generates its own id (e.g. UUIDv4 or ULID; format choice is an architect-stage detail). Caller-supplied `Idempotency-Key` headers are explicitly out of scope (see §Out of scope).
- **FR-11**: Opaque request/response pass-through. The proxy MUST treat the inbound caller request body and the upstream response body as opaque byte streams from the perspective of forwarding. The proxy MUST NOT deserialize either body into typed structures that would reject unknown fields; MUST NOT validate the JSON schema of either body; MUST NOT add, remove, rename, or coerce fields on either side. Limited inspection is permitted only for the narrow purposes explicitly required by other FRs (FR-4 stream detection; FR-8 size check; FR-12 Content-Type check). Upstream `Content-Type` (e.g. `application/json` or, in error scenarios, `text/html` from a CDN, `text/plain` from an edge) MUST be forwarded to the caller verbatim; the proxy MUST NOT rewrap upstream non-JSON bodies into a synthetic JSON envelope. This explicit schema-agnostic policy is the proxy's defence against upstream schema drift: when DeepSeek adds, removes, or changes fields, the proxy continues to forward bytes unchanged and callers handle the drift.
- **FR-12**: HTTP surface strictness.
  - Method strictness: a request with HTTP method other than `POST` on path `/v1/chat/completions` MUST be rejected with HTTP **405 Method Not Allowed** and an `Allow: POST` header.
  - Path strictness: a request to any path other than `/v1/chat/completions` on the API port (and `/metrics`, `/healthz` on the metrics port — exact paths an architect-stage choice) MUST be rejected with HTTP **404 Not Found**.
  - Content-Type strictness: a `POST /v1/chat/completions` request whose `Content-Type` header is absent or is not `application/json` (case-insensitive media-type match; charset parameter ignored) MUST be rejected with HTTP **415 Unsupported Media Type** before any upstream call. The proxy does NOT parse the JSON body to validate it (per FR-11); malformed JSON within an `application/json` request is forwarded upstream and DeepSeek's response (likely 400) is forwarded back.
  - TLS verification: the proxy MUST verify upstream TLS certificates against the system trust store. The proxy MUST refuse to start (exit non-zero on boot, with a clear error) if `NODE_TLS_REJECT_UNAUTHORIZED` is set to `0` or `false` in its environment. Other TLS hardening (pinning, MTLS) is out of scope.

> Each FR has a stable ID (FR-N). Reviews and ADRs reference these IDs.

## Non-functional requirements

### Latency

- **NFR-LAT-1**: Total wall-time per logical caller request, measured from the moment the proxy begins handling the inbound request to the moment it begins writing a response to the caller, MUST NOT exceed **30 seconds**. This budget includes: the initial upstream attempt, any jittered or `Retry-After`-bounded backoff sleep (per FR-6), and any retry attempt. If the budget is exceeded before a usable upstream response is available, the proxy MUST abort further work and respond to the caller with HTTP **504 Gateway Timeout** and a JSON error body. **Abort propagation (mandatory)**: at the moment the wall-time guard fires, the proxy MUST cancel any in-flight upstream HTTP request for that logical call — closing the upstream socket / aborting the fetch controller — and MUST NOT consume any further upstream response bytes. Continuing to read upstream bytes after returning 504 to the caller would mean the proxy is still spending DeepSeek quota and money on a request the caller already considers failed; this is explicitly forbidden. Sizing rationale: per-attempt budget (NFR-LAT-2) is 12 s, jittered backoff (FR-6) is up to 0.6 s, two attempts → worst case 12 + 0.6 + 12 = 24.6 s, comfortably within 30 s with ~5.4 s headroom for proxy-internal overhead (TLS resume, GC pauses, event-loop scheduling, body buffering).
- **NFR-LAT-2**: Per-attempt upstream timeouts (each individual attempt to DeepSeek, applied independently to the initial attempt and to the retry attempt under FR-6). **TCP connect timeout: 3 seconds** (from socket() to established TCP+TLS). **First-byte (time-to-first-byte) timeout: 12 seconds** (from request bytes flushed to first response byte received from upstream). **Total per-attempt timeout: 12 seconds** (hard cap on a single attempt's wall-time, inclusive of connect + send + first byte + body read). Exceeding the connect timeout or the first-byte timeout before any response bytes have been received counts as a Class A transport-level failure for the purposes of FR-5 and is eligible for retry within the FR-6 budget. Exceeding the total per-attempt timeout AFTER response bytes have begun arriving is NOT retried (per FR-5 cutoff) and surfaces as 504 to the caller via the NFR-LAT-1 abort path if the wall-time also expires; if the wall-time has not expired, the partial response is treated as a transport-mid-body failure and surfaces as `transport_error` with HTTP 502 to the caller (no retry).

### Throughput

- **NFR-THR-1**: Sustained throughput target — **5 RPS** with a **burst of 5** — bounded by and equal to the outbound token-bucket configuration in NFR-CAP-1. The proxy does not target a higher throughput than the outbound cap allows; inbound bursts that would exceed the available outbound tokens are rejected immediately per FR-7 (429 + `Retry-After` + `X-RateLimit-Reset-Ms`), not queued. There is no separate inbound throughput throttle.
- **NFR-CAP-2**: Expected inbound load profile (forecast, not a guarantee). **Sustained inbound rate: < 1 RPS** under normal operating conditions. **Peak inbound rate: < 3 RPS** during short-lived bursts. **Growth horizon: 12 months, flat** — no organic growth in caller traffic is expected over the next year; this profile is the planning baseline for the iteration. Relationship to NFR-CAP-1: the configured outbound cap (5 RPS, burst 5) leaves a comfortable margin above expected peak inbound (3 RPS), so FR-7 is expected to fire only on anomalous traffic spikes well above the forecast envelope, not in steady-state operation.

### Availability / SLA

- **NFR-AVL-1**: Best-effort availability. The proxy makes **no monthly availability commitment** (no 99% / 99.9% / 99.95% SLO) in this iteration. Rationale: single-instance deployment (per NFR-DEP-2) cannot honestly support a multi-nine SLO, and the proxy is an internal tool behind a perimeter-trusted network (NFR-SEC-1) with low forecast traffic (NFR-CAP-2). Operational expectations the proxy MUST meet:
  - **Graceful shutdown on SIGTERM** with **drain window ≥ NFR-LAT-1 wall-time**: on receiving SIGTERM, the proxy MUST stop accepting new inbound caller requests immediately. New inbound connections during the drain MUST be rejected with HTTP **503 Service Unavailable** and `Connection: close` (or, if the listener is already closed, a TCP refusal — operator's choice via the deployment configuration). In-flight requests at the moment of SIGTERM MUST be allowed to complete or hit their NFR-LAT-1 wall-time, so the drain window MUST be **at least 30 seconds** (i.e. ≥ NFR-LAT-1) to give every in-flight request a chance to finish gracefully. After the drain window, any remaining in-flight requests MAY be terminated. Configuring a shorter drain window in the deployment platform is a misconfiguration and SHOULD be caught by the operator's runbook (see NFR-DEP-2 deploy procedure).
  - **Process restart on crash**: the deployment platform's supervisor (Docker daemon `--restart=always` / Compose `restart: unless-stopped` per NFR-DEP-2) is expected to restart the proxy process automatically on crash or non-zero exit.
  - **No HA, no failover, no multi-region, no active-passive standby** in this iteration. A single instance going down means downtime until the supervisor restarts it; callers are expected to tolerate this.

### Durability

- **NFR-DUR-1**: The proxy is **stateless**. It holds no persistent state across process restarts: no database, no on-disk queue, no cache file, no journal. In-flight requests at the moment of crash or SIGTERM are handled per NFR-AVL-1's drain contract. **RPO (Recovery Point Objective) = N/A** — there is no committed data to recover. **No data-loss model is required** for this iteration; durability is intentionally not a concern of this service.

### Security

- **NFR-SEC-1**: The proxy MUST NOT be exposed to untrusted networks. Deployment assumes a closed network / VPC / service mesh with perimeter trust; the proxy itself enforces no caller authentication or authorization. Any client able to reach the proxy on the network is treated as authorized. **Secret-in-logs prohibition**: the upstream DeepSeek API key (sourced from `DEEPSEEK_API_KEY`, see NFR-DEP-2) MUST NOT appear in any stdout/stderr log line — ever, on any code path, including error/diagnostic paths. NFR-OBS-1 already forbids logging request/response bodies and sensitive headers; this constraint extends that prohibition explicitly to the API key value, so that incidental error-handling code (e.g. `console.log(error)` dumping a serialized request object that includes the `Authorization` header) does not leak the secret. Implementers MUST scrub or redact any log site that could plausibly serialize a request object, headers map, or error containing the key.
- **NFR-SEC-2**: Threat model — explicit acceptance and minimum controls for perimeter-trusted operation. **Accepted risks** (named, not fixed in this iteration):
  - Anyone reachable on the perimeter can drain the upstream DeepSeek key's quota by simply issuing requests to the proxy. There is no per-caller quota, no per-caller authentication, no per-caller billing attribution. A misbehaving internal CI job or compromised service can spike the DeepSeek bill until an operator stops the proxy. The team accepts this in exchange for the operational simplicity of "no caller auth"; mitigation is monitoring (NFR-OBS-1 outcome counters + DeepSeek bill alarms operator-side, not in this spec) plus the ability to stop the proxy quickly.
  - The shared upstream DeepSeek API key is a single point of compromise: any process with access to the proxy's environment (host, container) can read it from `/proc/<pid>/environ`, `docker inspect`, etc. The team accepts the container host as part of the trust boundary.
  - Logs identify callers by **source IP** and **`User-Agent`** (see NFR-OBS-1); these are NOT PII (no prompt content, no user identifiers), but they are the only attribution available in an incident. If two callers share a source IP (NAT, service mesh sidecar), they are indistinguishable in logs.
  - **Minimum control mandated**: the proxy MUST log the inbound source IP and `User-Agent` for every request (per NFR-OBS-1) so that an incident can be attributed to a host or service even without caller auth. Not logging these is treated as a regression.
  - Adding caller authentication (shared bearer token / mTLS / per-caller API keys) is explicitly out of scope (see Out of scope) but is the obvious next iteration if operating outside a strictly closed perimeter.

### Observability

- **NFR-OBS-1**: Minimal observability profile for this iteration.
  - **Logging**: structured JSON logs written to **stdout** at levels INFO / WARN / ERROR. Exactly **one log line per inbound caller request**, emitted at request completion, with the following fields:
    - `request_id` — proxy-assigned correlation id (string), also returned to the caller in `X-Request-Id` (FR-10).
    - `status` — the HTTP status code returned to the caller (integer).
    - `latency_ms` — caller-visible end-to-end latency (integer milliseconds).
    - `upstream_latency_ms` — sum of wall-time spent on upstream attempts (initial + retry, if any), excluding jittered backoff sleep (integer milliseconds; 0 if no upstream call was made, e.g. FR-7 local 429).
    - `upstream_status` — upstream HTTP status code returned by DeepSeek for the final attempt (integer; null if no upstream response was received, i.e. transport-only failure paths).
    - `retry_count` — number of retries actually performed (0 or 1, per FR-6).
    - `outcome` — closed enum, exactly one of: `ok`, `rate_limited` (FR-7 local-overflow 429 OR upstream-429 forwarded), `gateway_timeout` (NFR-LAT-1 wall-time exceeded → 504), `upstream_5xx` (DeepSeek returned 5xx other than 502/503-after-retry-exhausted, forwarded as-is — 502/503 after retry-exhaustion also fall here), `transport_error` (Class A transport failures from FR-5 exhausted retries; surfaced as 502 to caller), `client_error` (4xx returned to caller — covers FR-4 stream-rejection 400, FR-8 413, FR-12 405/404/415, and upstream 4xx forwarded).
    - `source_ip` — inbound TCP peer address as observed by the proxy (string; behind a TLS-terminating load balancer this is the LB's IP unless `X-Forwarded-For` honouring is added — out of scope here).
    - `user_agent` — value of inbound `User-Agent` header verbatim, or empty string if absent (string).
  - **PII / payload safety**: the proxy MUST NOT log inbound request bodies, outbound upstream request bodies, upstream response bodies, or caller response bodies. Inbound and upstream `Authorization` headers MUST NOT be logged. Other request/response headers SHOULD NOT be logged unless they appear in this NFR's explicit field list. This is an explicit safety choice: chat prompts and completions are treated as sensitive.
  - **Metrics endpoint**: served on a **separate TCP port** (default `9090`, see FR-9), bound to the same network the API listens on. Path: `/metrics`. The metrics port is intended for in-perimeter Prometheus scraping; the same NFR-SEC-1 perimeter-trust assumption applies (no auth on metrics).
  - **Required metrics** in this iteration:
    - `proxy_requests_total{outcome}` — counter of completed caller requests, labelled by the closed `outcome` enum above. No other labels.
    - `proxy_upstream_status_total{status}` — counter of upstream HTTP responses, labelled by the upstream HTTP status code as a string (e.g. `"200"`, `"429"`, `"502"`). Cardinality is bounded by the universe of HTTP status codes (low). Increments only when an upstream response is received; transport-only failures do not increment this counter.
    - `proxy_request_duration_ms` — histogram of caller-visible end-to-end latency (same value as `latency_ms`), unlabelled. Buckets MUST cover the relevant range; the architect MUST choose a bucket layout that includes at least: 1, 10, 100, 250, 500, 1000, 2000, 5000, 10000, 12000, 30000 (milliseconds).
    - `proxy_up` — gauge, value `1` while the proxy is healthy and serving traffic, `0` during shutdown drain (NFR-AVL-1).
    - `proxy_build_info{version, git_sha}` — info gauge (always `1`), with build identifier labels for liveness alerting and version-pinning.
  - **Tracing**: no distributed tracing in this iteration. See Out of scope.

### Capacity

- **NFR-CAP-1**: The proxy MUST enforce an **outbound** rate cap on its own request flow to DeepSeek (proxy throttles itself before issuing upstream calls) to avoid triggering upstream 429 storms. The cap is implemented as a **token bucket** with a **steady-state rate of 5 requests/second** and a **burst capacity of 5 tokens**. The cap is **process-wide** within a single proxy instance and applies to all outbound DeepSeek calls regardless of caller. There is no inbound rate limit applied to callers — the proxy does not police caller request rates. Overflow behavior (rejection semantics when the bucket is empty) is specified in FR-7. **Single-instance assumption**: the cap is per-process, so running N concurrent proxy instances multiplies the effective outbound rate to N × 5 RPS. Single-instance enforcement is delegated to FR-9 (port-bind-or-die) plus NFR-DEP-2's stop-then-start deploy procedure.

### Dependencies

- **NFR-DEP-1**: External dependencies and runtime baseline.
  - **Upstream provider**: DeepSeek REST API. Default base URL `https://api.deepseek.com`, overridable at deployment time via an environment variable (e.g. `DEEPSEEK_BASE_URL`) to support staging/test endpoints or a local mock; the default MUST be the production DeepSeek endpoint.
  - **Upstream auth**: HTTP header `Authorization: Bearer <DEEPSEEK_API_KEY>`, where `DEEPSEEK_API_KEY` is supplied per NFR-DEP-2 and protected per NFR-SEC-1.
  - **Runtime**: Node.js LTS, specifically version **v20 or v22**. Older or non-LTS Node versions are not supported.
  - **OS / architecture**: linux/amd64 (the Docker base image and CI build target).
  - **No fallback provider**: if DeepSeek is unreachable, requests fail per FR-5 (transport-level retry exhausted → `transport_error` outcome, 502 to caller) or surface upstream HTTP responses verbatim. Routing to an alternative LLM provider is out of scope (see §Out of scope).

### Deployment

- **NFR-DEP-2**: Deployment topology, secret delivery, and deploy procedure.
  - **Runtime**: a single Node.js process running inside a Docker container. One container = one proxy instance.
  - **Supervisor**: the Docker daemon. The container MUST be started with a restart policy equivalent to `--restart=always` (plain `docker run`) or `restart: unless-stopped` / `restart: always` in Docker Compose. No external supervisor (systemd, Kubernetes, Nomad) is assumed in this iteration.
  - **Configuration transport**: all runtime configuration is passed via **environment variables** at container start. No config files, no config server, no runtime mutation.
  - **Secret delivery**: the upstream DeepSeek API key is supplied as the `DEEPSEEK_API_KEY` environment variable. There is no integration with a secrets manager in this iteration. Operator responsibilities: (a) keep the source of `DEEPSEEK_API_KEY` (e.g. `.env` file, CI variable, host shell export) out of version control, (b) avoid surfaces that echo container env to other tenants (shared `docker inspect`, unrestricted `ps`/`/proc`), (c) rotate the key by restarting the container with a new value. The proxy itself enforces NFR-SEC-1's secret-in-logs prohibition on the runtime side.
  - **Single-instance enforcement**: the deployment MUST guarantee at most one proxy container is running at any moment. Mechanisms (any one is sufficient):
    - On a single Docker host: map a fixed host port to the container's API port (e.g. `-p 8080:8080`) — a second container attempting the same port mapping fails immediately due to host-port conflict; combined with FR-9's bind-or-die behaviour, this provides automatic enforcement.
    - On Docker Compose: declare exactly one service with `replicas: 1` (or omit the field; default is 1) and use a stop-then-start deploy procedure (see below).
    - On any other platform (k8s, Nomad, ECS): out of scope for this iteration; if the operator chooses to deploy on such a platform, they MUST replicate the single-replica + stop-then-start guarantee themselves.
  - **Deploy procedure (mandatory, runbook-grade)**: deploys MUST follow **stop-then-start**, NOT rolling overlap. Specifically:
    1. Pull the new image: `docker compose pull` (or equivalent).
    2. Stop the running container and wait for it to exit: `docker compose stop` + `docker compose wait` (or check `docker ps`); during this step the proxy executes its NFR-AVL-1 graceful drain.
    3. Start the new container: `docker compose up -d`.
    4. Verify by hitting the metrics endpoint `/metrics` on the metrics port and confirming `proxy_up == 1`.
    Operators MUST NOT use Docker Compose's default `up -d` rolling behaviour (which can briefly run new + old together while the new container starts), MUST NOT use `docker compose up --no-recreate`, and MUST NOT scale `replicas` above 1 even temporarily. During the stop-then-start window callers WILL see brief unavailability (TCP refusal or 503 per NFR-AVL-1); this is an accepted property of the deployment, not a bug.
  - **Container resource limits and Node heap settings**: explicit limits are NOT mandated by this spec — left to the operator (a sensible starting point is container memory limit ≥ 512 MiB and Node `--max-old-space-size=384`, but these numbers are not load-bearing on the spec). The combination of FR-8 (256 KiB body cap) and NFR-CAP-1 (5 RPS outbound) bounds the worst-case in-memory body buffering at well under 100 MiB even with all concurrent attempts buffered, so default Node limits are sufficient for the forecast load.

## Out of scope

- Multi-provider support (OpenAI, Anthropic, etc.). This iteration targets DeepSeek only.
- Caller authentication and authorization (shared bearer token, mTLS, per-caller API keys, per-caller quotas). The trust boundary is the network perimeter (closed network / VPC / service mesh); see NFR-SEC-1 and NFR-SEC-2.
- Other DeepSeek endpoints (text completions, embeddings, models list, FIM/beta endpoints, etc.) — not exposed in this iteration. Only `POST /v1/chat/completions` is surfaced (FR-2).
- SSE / streaming pass-through (`"stream": true`) — deferred to a future iteration. See FR-4.
- Distributed tracing — deferred. No OpenTelemetry spans, no W3C `traceparent` propagation. Observability is limited to NFR-OBS-1's stdout-JSON-logs + Prometheus metrics profile.
- Logging of request/response bodies, headers (other than `User-Agent`), and any prompt/completion content — explicitly excluded for PII safety. See NFR-OBS-1.
- Caller-supplied `Idempotency-Key` headers and proxy-side request deduplication. The proxy generates its own `request_id` (FR-10) and ignores caller-supplied idempotency-style headers. Known consequence: if a caller's TCP fails after the proxy has flushed bytes upstream, the caller has no safe way to retry without risking double billing — callers MUST treat this as a known failure mode (acknowledged risk, not a defect).
- Capturing upstream `x-request-id` / `cf-ray` / similar headers from DeepSeek responses for cross-correlation in proxy logs. Useful for incident triage but deferred (minor objection #11 in spec-review.md).
- Operator-configurable values for FR-8 body cap, FR-9 ports, NFR-CAP-1 bucket parameters, and similar. All numbers are fixed for this iteration; making them configurable is an architect-stage or next-iteration concern.
- Honouring `X-Forwarded-For` / `Forwarded` headers to record the real caller IP behind a load balancer. Source IP in NFR-OBS-1 is the direct TCP peer; LB-aware logging is a future iteration.
- Multi-replica / horizontal scaling. NFR-CAP-1's bucket is process-local; running ≥2 replicas is a correctness violation, not a scaling option.
